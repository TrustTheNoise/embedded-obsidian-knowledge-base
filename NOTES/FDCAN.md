# Блок схема
![[Pasted image 20250402143730.png]]
`fdcan_clk` - тактовый сигнал для FDCAN, источник которого можно выбрать при помощи мьютекса
![[Pasted image 20251004155243.png|300]]
Источник выбирается в поле FDCANSEL[1:0] в регистре **RCC_CCIPR** или через функцию `__HAL_RCC_FDCAN_CONFIG()`. Подробнее написано в [[#Тактирование для FDCAN]]


`fdcan_tq_ck` - это тактовый сигнал, который используется для синхронизации временных квантов в блоке FDCAN. Он генерируется на основе основного ОТДЕЛЬНОГО ТАКТОВОГО СИГНАЛА fdcan_clk для FDCAN источником которого можно выбрать при помощи мьютекса. Именно fdcan_tq_ck генерирует тики для работы CAN core.


`fdcan_ts[0:15]` - 16-битный генератор временных меток, который прикрепляется к входящим (и иногда исходящим) кадрам. При получении сообщения RX Handler автоматически фиксирует время его прибытия с точностью до одного tq(тика).

`CAN core` - центральный элемент блока FDCAN, который управляет передачей и приемом данных по шине CAN, также именно он собирает [контрольную сумму CRC](CAN%20шина#Контрольная%20сумма%20кадра(CRC))  и добавляет [bit stuffing](CAN%20шина#Дополняющие%20биты%20(bit%20stuffing)). Он синхронизуется с тактовым сигналом `fdcan_tq_ck` и взаимодействует с другими блоками для обработки данных.

`TX Handler` - отвечает за подготовку и упорядочивание сообщений для отправки. Он может работать как FIFO(т.е. первый зашёл, первый вышел), так и как очередь со своими правилами. Например он берёт данные из `Message RAM` смотрит на идентификаторы этих сообщениях, расставляет эти [идентификаторы по приоритету](CAN%20шина#Арбитраж) и вот таким образом TX Handler собирает очередь из сообщений. В TX event - FIFO записывает после каждого сообщения временную метку в тиках

`RX Handler` - Этот блок занимается обработкой входящих сообщений. Он включает механизм фильтрации – настраиваемые фильтры позволяют отсеивать нежелательные кадры, принимая только те, что удовлетворяют заданным условиям (например, совпадению идентификатора или определённой маске). Поддерживается до 28 фильтров для 11-битных идентификаторов и до 8 фильтров для 29-битных идентификаторов. RX Handler поддерживает два Rx FIFO для хранения всех сообщений, что позволяет разделить потоки сообщений или обеспечить дополнительный запас для хранения входящих данных, чтобы не потерять сообщения при высокой загрузке.

`Message RAM interface` - Для выбора какие сообщения нужно передать, куда передавать полученные сообщения(например, FIFO для входящих кадров), а также для настройки фильтров приёма используется область памяти в RAM которая и называется `Message RAM`. В Message RAM организуются отдельные буферы для хранения кадров, что позволяет модулю работать независимо от центрального процессора (CPU). Собственно говоря `Message RAM interface` и позволяет работать с этой памятью

`Control and Configuration Registers` - регистры для настройки самой периферии FDCAN

`Interrupts Interface` - этот блок генерирует прерывания. Причём есть две линии для прерываний `fdcan_intr1_it` и `fdcan_intr2_it`. Это сделано для снижения нагрузки на обработчик прерывания. Ведь если у нас будет поток событий на одну линию, то если у нас будет много прерываний одновременно, при этом с разными приоритетами, то у нас будет "узкое место"

# Тактирование для FDCAN
У тактирования FDCAN есть особенность. Да настройка FDCAN через регистры и работа с [[#Message RAM]] осуществляется с тактированием шины APB, однако в [[#Блок схема]] можно заметить что основа для создания tq это fdcan_clk, который независим от APB. Соответственно его можно настраивать через регистр RCC_CCIPR периферии [RCC](RCC.md). И тут самое интересное, у того же [I2C] где тоже есть независимое тактирование i2c_ker_ck которое также настраивается через RCC_CCIPR, но по умолчанию для I2C выбран именно SYSCLK. НО! Для FDCAN же по умолчанию выбран HSE, а SYSCLK как и HSI нельзя выбрать в принципе
![[Pasted image 20250607182807.png]]
Я не смог найти этот момент в reference manual но я помню что там про это было написано что мол HSI имеет низкую точность, поэтому в stm32 ЗАПРЕЩЕНО использовать HSI как источник тактирования, ведь в CAN очень важно небольшое отклонение, потому что не всё может покрывать ресинхронизация в [[#Bit timing]], максимальное толлерантность к отклонению стандартного CAN 1.58%, а HSI обычно имеет отклонение в 1%(или хуже без калибровки).
Соответственно нельзя выбрать источником tq HSI а так как как SYSCLK может быть тот же HSI то нельзя выбрать и SYSCLK как источник.
> [!warning] 
> Как источник PCLK может быть выбран тот же HSI, что сделает точно также большую неточность. Так что использовать PCLK нужно только в случае если его источником выбран HSE! 

# Message RAM
В первую очередь мы тут для чего? Правильно, чтобы передавать и принимать через CAN шину! Так вот. Если в интерфейсах типа [[I2C]] или [[SPI]] вся работа с передачей и приёмом происходила через регистры, то в FDCAN работа с передачей/приёмом выполняется с помощью отдельно выделенного участка RAM - с помощью Message RAM. Message RAM в ширину 32 бит, и состоит из 212 слов(однако не обязательно настраивать их все)

## Адресация Message RAM
FDCAN обращается к Message RAM пословно, а не побайтно. 
В stm32 есть несколько FDCAN соответственно для каждого выделены свои адреса в Message RAM, причём адресация для каждого FDCANn идёт по порядку.
- FDCAN1:
	- начальный адрес: 0x0000
	- конечный адрес: 212(слов в message ram)*4(байтов в 1 слове)=848=0x350
- FDCAN2:
	- начальный адрес: конечный адрес FDCAN1 0x350+0x4=0x354
	- конечный адрес: 0x354+0x350 - 0x4 = 0x6A0

# Фильтрация приёма
Структура для FDCAN фильтров такая
![[Pasted image 20251004161157.png]]
![[Pasted image 20251004161205.png]]

В 11-bit и 29-bit фильтрах можно выбрать 2 ID в фильтре: SFID1, SFID2 для 11 битных фильтров и EFID1, EFID2 для 28 битных фильтров. Также выбрать тип фильтра с этими двумя ID через биты SFT[1:0] и EFT[1:0] и конфигурацию каждого из элементов фильтра SFEC[2:0]/EFEC[2:0]

Глобальные настройки фильтров находятся в регистре **FDCAN_RXGFC**. В них есть настройки:
1. Количества фильтров. Для 11 битных фильтров(LSS) их может быть до 28, для 29 битных фильтров(LSE) - до 8. Соответственно не обязательно их все использовать. 
> [!caution] 
>  Проверка полученного Message ID под каждый фильтр происходит последовательно по порядку.

2. Выбор как поступать с не подходящими фильтру значению. Можно записывать их в один из RX FIFO либо отбрасывать.

В зависимости от конфигурации элемента фильтра (SFEC/EFEC), совпадение полученного сообщение с фильтром вызывает одно из следующих действий:

- Сохранить полученный фрейм в FIFO 0 или FIFO 1.
- Отклонить полученный фрейм.
- Установить флаг прерывания для сообщений высокого приоритета IR[HPM].
- Установить флаг прерывания для сообщений высокого приоритета IR[HPM] и сохранить полученный фрейм в FIFO 0 или FIFO 1.

Итак самой же фильтрации есть несколько вариантов, о них и поговорим ниже

### Фильтр диапазона(FT=00)
Фильтр соответствует всем полученным фреймам с Message ID в диапазоне, определенном SF1ID/SF2ID и EF1ID/EF2ID.
Существует две варианта фильтра диапазона для 29 битных ID:
- **EFT = 00**: Message ID полученных фреймов выполняет операцию AND с расширенной маской XIDAM(который пользователь выбирает в регистре FDCAN_XIDAM) а затем проверяет полученное на фильтре диапазона
> [!example] 
>  Пусть XIDAM=0x7FF (маска, которая оставляет только 11 младших битов), EF1ID=0x123, EF2ID = 0x456.
>  Мы получили сообщение с ID=0xABC
>  Сначала мы делаем AND с XIDAM
>  0xABC & 0x7FF = 0x2BC
>  И уже полученное проверяем попадает ли в диапазон от `0x123` до `0x456`. В нашем случае да, это сообщение попадает под фильтр

- EFT = 11: Расширенная маска AND (XIDAM) не используется для фильтрации диапазона.

### Фильтр для определённых ID(FT=01)
Элемент фильтра может быть настроен для фильтрации одного или двух конкретных Message ID. То есть первый ID будет в F1ID, а второй ID в F2ID. Для фильтрации одного конкретного Message ID элемент фильтра должен быть настроен с SF1ID = SF2ID и EF1ID = EF2ID.

### Масочный фильтр(FT=11)
Классическая битовая масочная фильтрация предназначена для фильтрации групп Message ID путем маскирования отдельных битов полученного Message ID. При классической битовой масочной фильтрации SF1ID/EF1ID используется как фильтр Message ID, а SF2ID/EF2ID используется как маска фильтра.

Идея в чём. Только биты с 1 в маске фильтра имеют значение для фильтра, тогда как биты с 0 в маске фильтра игнорируются вовсе и считаются априори подходящими под фильтр.

А вот те биты в ID полученного сообщения соответствующие 1 в маске фильтра сравниваются с F1ID и если они равны, то в таком случае этот ID подходит под фильтр. Лучше показать на примере
> [!example] 
>  SF1ID=0b1100_1100, SF2ID=0b1111_1000 а в сообщении мы получили Message_ID=0b1100_1110
>  В итоге из-за маски мы должны сравнить 5 старших битов в Message_ID и SF1ID
>  Для SF1ID это 0b1100_1~~100~~
>  Для Message_ID это 0b1100_1~~110~~
>  Как можно заметить полученные 5 старших битов одинаковые, соответственно полученный Message_ID подходит под наш фильтр.

# Rx FIFO
Rx FIFO 0 и Rx FIFO 1 могут содержать до трех элементов каждый. Полученные сообщения, прошедшие [фильтрацию приема](#Фильтрация%20приёма), передаются в соответствующий Rx FIFO, как настроено фильтром(в SFEC/EFEC).

Сами же RX FIFO и TX FIFO представляют из себя circular buffer с довольно странным управлением
![[Circular_Buffer_Animation.gif|300]]
Итак у нас есть два регистра управления каждым RX FIFO: FDCAN_RXFnS и FDCAN_RXFnA. FDCAN_RXFnS является регистром только для чтения в котором у нас есть номер элемента записи в RX FIFO(`FnPI`) и номер элемента чтения из RX FIFO(`FnGI`), оба этих индексов только на чтение, также уровень заполненности RX FIFO(`FnFL`). В FDCAN_RXFnA же находится "acknowledge index" `FnAI` в который мы как раз таки записываем индекс последнего прочитанного элемента из RX FIFO. А уже это устанавливает индексы FnGI = FnAI+1 и обновляет значение FnFL.
Сделано управление RX FIFO слегка странно. Итак размер каждого элемента в RX FIFO фиксирован. То есть считается что у нас данных 64 байта(как будто мы настроены в CAN FD)
![[Pasted image 20250602114911.png]]

## Режимы переполнения для RX FIFO 0 и 1.
Если бит FnOM в регистре FDCAN_RXGFC сброшен. В случае, если сообщение получено, когда соответствующий Rx FIFO заполнен, это сообщение отбрасывается, и состояние потери сообщения сигнализируется установкой RXFnS.RFnL = 1. Также устанавливается флаг прерывания IR.RFnL. 
Когда достигается состояние полного заполнения Rx FIFO (RXFnS.FnPI = RXFnS.FnGI), никакие дальнейшие сообщения не записываются в соответствующий Rx FIFO, пока хотя бы одно сообщение не будет прочитано и индекс получения Rx FIFO не будет увеличен. 
Состояние полного заполнения Rx FIFO сигнализируется установкой RXFnS.FnF = 1. Кроме того, устанавливается флаг прерывания IR.RFnF.


Если же бит FnOM установлен в 1 то в случае когда сигнализируется состояние полного заполнения Rx FIFO (RXFnS.FnPI = RXFnS.FnGI) с помощью RXFnS.FnF = 1, следующее сообщение, принятое для FIFO, перезаписывает самое старое сообщение в FIFO. Индексы записи и получения увеличиваются на единицу.
> [!caution] 
>  Если у RXFnS.FnPI = RXFnS.FnGI т.е. FIFO заполнен и при этом работает режим перезаписи(FnOM=1). То может произойти такое, что у нас одновременно происходит запись в элемент с индексом FnPI и мы можем начать считывать данные с элемента с индексом FnGI, которые одинаковые. В этом случае могут быть прочитаны некорректные данные из соответствующего элемента Rx FIFO. Поэтому желательно считывать с элемента с индексом FnGI+1 когда поднят флаг RXFnS.FnF

## Генерация временных меток
Для генерации временных меток FDCAN предоставляет 16-битный счетчик с переполнением. Выбора какими будут временные метки можно выбрать в TSCC.TSS[1:0], временные метки могут быть:
- TSS = 0b00 - метки постоянно равны 0
- TSS = 0b01 - метки выставляются в соответствии с внутренним счётчиком TCP от FDCAN
- TSS= 0b10 - метки выставляются от счётчика TIM3 т.е. от tim3_cnt[0:15]

Если временные метки формируются от внутреннего счётчика TCP модуля FDCAN, то предделитель TSCC.TCP определяет, как часто счётчик временных меток увеличивается — один раз на каждые (TCP + 1) [bit time](CAN шина#Bit timing) CAN. Значение счетчика можно прочитать через TSCV[TCV]. Запись в регистр TSCV сбрасывает счетчик в 0. Когда счетчик временных меток переполняется, устанавливается флаг прерывания IR[TSW].

При начале приема кадра значение счетчика захватывается и сохраняется в секцию временной метки элемента Rx FIFO (RXTS[15:0]).

## Буфер передачи
Существуют два варианта буферизации передачи:
- TX FIFO - стандартный "First In, First Out", сообщения передаются в **точно том порядке**, в котором вы их положили в FIFO.
- TX queue(читается ["куэуэ"](https://www.youtube.com/watch?v=lcsXGHl_hwg)) - сообщения выбираются для передачи **по порядку приоритетов CAN-ID**: из всех записанных фреймов аппарат возьмёт сначала тот, у которого наименьший идентификатор.

Выбор между этими двумя режимами происходит через настройку бита TFQM в регистре **FDCAN_TXBC**(0 - TX FIFO, 1 - TX queue)

### TX FIFO(TFQM=0)
Сообщения, хранящиеся в Tx FIFO, передаются, начиная с сообщения, на которое ссылается индекс чтения TXFQS[TFGI]. После каждой передачи индекс получения увеличивается циклически, пока Tx FIFO не станет пустым(TXFQS[TFQF]=1 и TXFQS[TFFL]=0). FDCAN вычисляет уровень свободного пространства Tx FIFO TXFQS[TFFL] как разницу между индексами получения и записи т.е. TFGI-TFQPI.

> [!important] 
>  Tx FIFO позволяет передавать сообщения с одинаковым идентификатором сообщения из разных буферов Tx в порядке, в котором эти сообщения были записаны в Tx FIFO.

Новые передаваемые сообщения должны быть записаны в Tx FIFO, начиная с буфера Tx, на который ссылается индекс записи TXFQS[TFQPI]. Запрос на добавление увеличивает индекс записи до следующего свободного элемента Tx FIFO. Когда индекс записи достигает индекса чтения, сигнализируется, что Tx FIFO заполнен (TXFQS[TFQF] = 1). В этом случае никакие дальнейшие сообщения не должны записываться в Tx FIFO, пока не будет передано следующее сообщение и индекс получения не будет увеличен.

Когда у нас в TX FIFO добавляется хотя бы одно сообщение, передача запрашивается записью в AR[2:0] в регистре FDCAN_TXBAR. В AR[2:0] для каждого элемента TX buffer свой бит. Т.е. для запроса передачи для 1 элемента TX buffer нужно поднять AR[0], для 2 элемента AR[1] и т.д.

> [!help] 
> AR[2:0] это как флаг для FDCAN что мол хей, этот элемент в TX buffer настроен, запускай шарманку. 

> [!caution] 
>  Количество запрошенных буферов Tx(т.е. тех битов что выставлены в AR[2:0]) не должно превышать количество свободных буферов Tx, т.е. не должно быть больше чем TXFQS[TFFL].

Для того чтобы узнать какие из запрошенных ячеек **еще не отправлены**, то есть «ожидают передачи» можно посмотреть в регистр FDCAN_TXBRP. То есть в FDCAN_TXBAR мы только даём запрос для МК, и когда МК заметит этот запрос и поставит в очередь отправку он сбросит бит в FDCAN_TXBAR. Однако это ещё не значит, что этот буфер был передан, это только значит что МК поставил буфер в очередь.

Когда запрос на передачу для буфера Tx, на который ссылается индекс получения, отменяется, индекс получения увеличивается до следующего буфера Tx с ожидающим запросом на передачу, и уровень свободного пространства Tx FIFO пересчитывается.

### TX Queue(TFQM=1)
Сообщения, хранящиеся в очереди передачи, передаются, начиная с сообщения с наименьшим идентификатором сообщения (наивысший приоритет).

> [!important] 
> В случае смешивания стандартных и расширенных идентификаторов сообщений, стандартные идентификаторы сообщений сравниваются с битами [28:18] расширенных идентификаторов сообщений. 
> В случае, если несколько буферов очереди настроены с одинаковым идентификатором сообщения, сначала передается буфер очереди с наименьшим номером буфера.

Новые сообщения должны быть записаны в буфер Tx, на который ссылается индекс записи TXFQS[TFQPI]. Запрос на добавление циклически увеличивает индекс записи до следующего свободного буфера Tx. В случае, если очередь передачи заполнена (TXFQS[TFQF] = 1), индекс записи недействителен, и никакие дальнейшие сообщения не должны записываться в очередь передачи, пока хотя бы одно из запрошенных сообщений не будет отправлено или ожидающий запрос на передачу не будет отменен.

> [!hint] 
>  Когда мы используем TX queue можно использовать регистр FDCAN_TXBRP вместо индекса записи. FDCAN_TXBRP это регистр который показывает какие из запрошенных буферов в FDCAN_TXBAR ещё не были отправлены. А так как в TX queue не важен порядок куда мы всовываем данные, то можно записывать данные просто в те места, которые не были запрошены или уже были отправлены.

## Отмена передачи
FDCAN поддерживает отмену передачи. 
> [!warning] 
> Отмена передачи не предназначена для работы с режимом Tx FIFO для буфера!

Для отмены запрошенной передачи из TX queue хост должен записать 1 в соответствующую позицию бита (номер буфера передачи) регистра **FDCAN_TXBCR**. Успешная отмена сигнализируется установкой соответствующего бита регистра **FDCAN_TXBCF** в 1.

В случае, если запрос на отмену передачи поступает во время уже идущей передачи из буфера Tx, соответствующий бит TXBRP остается установленным до тех пор, пока передача не завершится. Если передача была успешной, устанавливаются соответствующие биты TXBTO и TXBCF. Если передача была неудачной, она не повторяется(если не установлен бит [CCCR[DAR]](#Режим%20отключенной%20автоматической%20повторной%20передачи(CCCR.DAR))), и устанавливается только соответствующий бит TXBCF.

> [!Tip] 
> В случае, если ожидающая передача отменяется непосредственно перед тем, как она могла бы быть начата, существует короткий временной интервал, в течение которого передача не начинается, даже если другое сообщение ожидает в узле. Это может позволить другому узлу передать сообщение, которое может иметь более низкий приоритет, чем второе сообщение в узле.

# Обработка подтверждения FIFO
Для поддержки обработки событий передачи FDCAN реализовал Tx event FIFO. После того, как FDCAN передал сообщение по шине CAN, идентификатор сообщения и временная метка сохраняются в элементе Tx event FIFO.

И уже в TX event можно просмотреть статус передачи этих данных. Разделение на отдельный раздел для обработки событий через TX event имеет большое преимущество, особенно при работе с динамически управляемой очередью передачи, так как буфер Tx может быть использован для нового сообщения сразу после успешной передачи. Нет необходимости сохранять информацию о статусе передачи из буфера Tx перед перезаписью этого буфера Tx.

Когда сигнализируется состояние полного заполнения Tx event FIFO с помощью IR[TEFF], никакие дальнейшие элементы не записываются в Tx event FIFO, пока хотя бы один элемент не будет прочитан и индекс получения Tx event FIFO не будет увеличен. В случае, если событие передачи происходит, когда Tx event FIFO заполнен, это событие отбрасывается, и устанавливается флаг прерывания IR[TEFL].

## Генерация временных меток
Для генерации временных меток FDCAN предоставляет 16-битный счетчик с переполнением. Выбора какими будут временные метки можно выбрать в TSCC.TSS[1:0], временные метки могут быть:
- TSS = 0b00 - метки постоянно равны 0
- TSS = 0b01 - метки выставляются в соответствии с внутренним счётчиком TCP от FDCAN
- TSS= 0b10 - метки выставляются от счётчика TIM3 т.е. от tim3_cnt[0:15]

Если метки выставляются в соответствии с внутренним счётчиком TCP от FDCAN то предделитель TSCC.TCP может быть настроен для тактирования счетчика в кратных битового времени CAN (от 1 до 16). Значение счетчика можно прочитать через TSCV[TCV]. Запись в регистр TSCV сбрасывает счетчик в 0. Когда счетчик временных меток переполняется, устанавливается флаг прерывания IR[TSW].

При начале передачи кадра значение счетчика захватывается и сохраняется в секцию временной метки элемента TXeFIFO (TXTS[15:0]).

# Счётчик тайм-аута
Для сигнализации условий тайм-аута для Rx FIFO 0, Rx FIFO 1 и FIFO событий передачи FDCAN предоставляет 16-битный счетчик тайм-аута. Он работает как счетчик обратного отсчета и использует тот же предделитель, управляемый TSCC[TCP], что и [счетчик временных меток](#Генерация%20временных%20меток)

Этот счётчик позволяет следить за тем, как долго в очереди на считывание(RX FIFO n) или передачу(TXeFIFO) есть сообщения и если они задерживаются слишком долго, выдать специальный флаг IR[TOO].

Текущее значение счетчика можно прочитать из TOCV[TOC].
> [!warning] 
> Счетчик тайм-аута может быть запущен только при CCCR[INIT] = 0. Он останавливается, когда CCCR[INIT] = 1, например, когда FDCAN переходит в состояние Bus_Off. 

Режим работы выбирается с помощью TOCC[TOS]:
- TOS = 0b00 - непрерывный режим
- TOS = 0b01 - тайм-аут контроллируется TX event FIFO
- TOS = 0b10 - тайм-аут контроллируется RX FIFO0
- TOS = 0b11 - тайм-аут контроллируется RX FIFO1

В непрерывном режиме счетчик начинает отсчет при сбросе CCCR[INIT]. Запись в TOCV предварительно устанавливает счетчик в значение, настроенное TOCC[TOP], и продолжает обратный отсчет. Когда он дойдёт до 0 установится флаг IR[TOO] и в TOCV загрузится значение TOCC[TOP].
Если до того, как он дошёл до 0, вы в программном коде напишете новое значение в `TOCV` то счётчик **сразу же** перезагрузится тем значением, которое вы записали (или получит заново `TOP`, если вы прямо в TOCV положили `TOP`). После этого продолжит уменьшаться дальше.

> [!attention] 
>  TOCV можно изменять только в непрерывном режиме TOS=0b00

Когда счетчик тайм-аута управляется одним из FIFO, пока выбранный FIFO пустой то в TOCV записывается значение TOCC[TOP]. Как только в выбранный FIFO поступило первое сообщение то счётчик начинает убывать вниз. То есть нужно чтобы FIFO снова был полностью пуст, в таком случае счётчик перезапустится с TOCC[TOP].
В случае если счётчик опустеет до 0, то установится флаг IR[TOO] и при этом счётчик останется на 0 до тех пор пока FIFO не опустеет, чтобы счётчик сбросился.


# Режимы работы FDCAN
## Режим ограниченного функционирования(CCCR.ASM)
В режиме ограниченного функционирования узел способен принимать данные и удаленные фреймы, а также подтверждать корректные фреймы, но он не отправляет данные, запросы на получение данных(RTR=1), активные фреймы ошибок или фреймы перегрузки.

В CAN сети, если узел начинает вести себя ненадежно (например, из-за ошибок передачи), он может быть переведён в режим ограниченного функционирования. Ведь если полностью отключить узел, то это может привести к потере важных функций этого узла. Однако чтобы узел не отправлял ошибочные пакеты в сети ему и отключают такую возможность. Таким образом в режиме ограниченного функционирования узел будет просто принимать данные и подтверждать приём. Это помогает сохранить общую стабильность сети, пока проблема узла не будет решена. Когда же проблема на узле решится можно перевести из режима ограниченного функционирования.

В режиме ограниченного функционирования счётчики ошибок (ECR.REC, ECR.TEC) замораживаются, в то время как регистрация ошибок (ECR.CEL) остается активной. В этот режим можно войти как программно так и автоматически.
- Чтобы перейти в него программно нужно установить бит CCCR.ASM. Этот бит может быть установлен программно только тогда, когда оба бита CCCR.CCE и CCCR.INIT установлены в 1. Бит может быть сброшен программно в любое время.
- Режим автоматически активируется, если обработчик передачи не смог прочитать данные из Message RAM вовремя. Чтобы выйти из режима ограниченного функционирования, программное обеспечение должно сбросить бит CCCR.ASM.

> [!example] 
>  Режим ограниченного функционирования может быть использован в приложениях, которые адаптируются к различным скоростям передачи данных CAN. В этом случае приложение тестирует различные скорости передачи и выходит из режима ограниченного функционирования после получения корректного фрейма.

> [!caution] 
>  Режим ограниченного функционирования не должен комбинироваться с режимом обратной связи (внутренней или внешней).

## Режим мониторинга шины(CCCR.MON)
Название говорит само за себя. Режим мониторинга работает как [[#Режим ограниченного функционирования(CCCR.ASM)]] с той лишь разницей, что в режиме мониторинга шины FDCAN отправляет только рецессивные биты на шину CAN. То есть если FDCAN должен отправить доминантный бит (бит подтверждения ACK, флаг перегрузки, активный флаг ошибки), бит перенаправляется внутренне, так что FDCAN может его мониторить, но при этом узел не отправляет доминантный бит для ACK и прочего на шину.

В режиме мониторинга шины регистр TXBRP удерживается в состоянии сброса.

![[Pasted image 20250601133722.png|250]]
> [!caution] 
>  Режим мониторинга шины включается установкой бита CCCR.MON CCCR.CCE и CCCR.INIT установлены в 1. Бит может быть сброшен программно в любое время.

> [!example] 
>  Режим мониторинга шины может быть использован для анализа трафика на шине CAN без воздействия на него передачей доминантных битов.

## Режим отключенной автоматической повторной передачи(CCCR.DAR)
По умолчанию автоматическая повторная передача включена. Соответственно FDCAN обеспечивает средства для автоматической повторной передачи фреймов, которые потеряли арбитраж или были нарушены ошибками во время передачи.

Чтобы отключить автоматическую повторную передачу нужно установить бит CCCR.DAR который может быть установлен и очищен в любой момент.

В режиме DAR (Discontinue Automatic Retransmission) все передачи автоматически отменяются после их начала на шине CAN. Бит ожидания запроса на передачу Tx буфера TXBRP.TRPx сбрасывается после успешной передачи или же любой ошибки, в том числе потери арбитража и т.д.
- **Успешная передача**:
    - Бит события передачи соответствующего Tx буфера TXBTO[TOx] установлен.
    - Бит завершения отмены соответствующего Tx буфера TXBCF[CFx] не установлен.
- **Успешная передача несмотря на отмену**:
    - Бит события передачи соответствующего Tx буфера TXBTO[TOx] установлен.
    - Бит завершения отмены соответствующего Tx буфера TXBCF[CFx] установлен.
- **Потеря арбитража или нарушение передачи кадра**:
    - Бит события передачи соответствующего Tx буфера TXBTO[TOx] не установлен.
    - Бит завершения отмены соответствующего Tx буфера TXBCF[CFx] установлен.

Если хранение событий передачи включено, тогда после успешной передачи кадра в соответствующем элементе TXeFIFO записывается ET=10(передача несмотря на отмену)

## Тестовый Режим(CCCR.TEST)
Для разрешения записи в регистр тестирования FDCAN_TEST необходимо установить бит CCCR.TEST в 1, что позволяет настраивать тестовые режимы и функции.

Тестовый режим нужен для отладки и тестирования аппаратной части CAN-сети(т.е. физических соединений, согласования, драйверов и т.д.).

В тестовом режиме через биты TEST[TX] можно установить чтобы выходной пин TX:
1. TX=00 - работал как обычно
2. TX=01 - выдавал 1 только в момент Sample point. Таким образом можно через логический анализатор понять насколько хорошо было настроен bit time
3. TX=10 - выдавал всё время 0
4. TX=11 - выдавал всё время 1

TEST[RX] показывает какое состояние на данный момент у FDCAN_RX 0 или 1.

У тестового режима есть ещё подрежим - режим обратной связи
### Внешняя обратная связь(TEST.LBCK)
В режиме внешней обратной связи FDCAN обрабатывает свои собственные переданные сообщения как полученные и сохраняет их (если они проходят фильтрацию приема) в Rx FIFO.
![[Pasted image 20250606154601.png|200]]
Этот режим предоставляется для аппаратной самопроверки. Переданные сообщения можно контролировать на выходном контакте передачи FDCAN_TX.
### Внутренняя обратная связь(TEST.LBCK и CCCR.MON)
Этот режим может использоваться для "горячей самопроверки", что означает, что FDCAN может быть протестирован без влияния на работающую систему CAN, подключенную к контактам FDCAN_TX и FDCAN_RX. В этом режиме контакт FDCAN_RX отключается от FDCAN, а контакт FDCAN_TX удерживается в рецессивном состоянии.

# FDCAN для CAN classic
Режим CAN classic для FDCAN выставляется очищением бита CCCR.FDOE, в таком случае полученные кадры интерпретируются как классические CAN кадры, что приводит к передаче кадра ошибки при получении CAN FD кадра. Когда работа CAN FD отключена, CAN FD кадры не передаются, даже если бит FDF элемента буфера передачи установлен, точно также игнорируются флаг BRS.

## Тайминги
Информацию по bit timing можно найти [тут](CAN%20шина#Bit%20timing)

Итак bit timing в stm32 выглядит таким образом
![[Pasted image 20250402152801.png]]

- Сам микроконтроллер ожидает что будет происходить изменение в SyncSeg, которая фиксированная и занимает 1 tq. Когда же МК отправляет данные на CAN шину, он это делает как раз таки в SyncSeg 
- После SyncSeg мы ожидаем время Bit segment 1(BS1) в конце которого мы сэмплируем полученный бит и записываем его в RX FIFO. BS1 состоит из Prop_Seg и Phase_Seg1 стандарта BOSCH.
> [!question] 
>  Возможно у знающего человека возникнет вопрос нету ли времени между сэмплированием и записыванием в RX FIFO
- Далее ждём Bit segment 2(BS2) в конце которого мы передаём уже наше значение из TX buffer.
Тайминг для одного бита в пакете представляет из себя сумму времени занимаемой sync_seg, bs1 и bs2

$$bit\ time = t_{SyncSeg}+t_{BS1}+t_{BS2}$$
То есть нам нужно чтобы $\large \frac{1}{bit\ time}=$ нужная нам частота CAN.

bit timing состоит из tq и все временные задержки рассчитываются по такой формуле
$$tq = (FDCAN\_NBTP.NBRP[8:0] + 1) \cdot t_{fdcan\_tq\_clk}$$
$$t_{SyncSeg}=1\ tq$$
$$t_{BS1}= tq\ \cdot \  (FDCAN\_NBTP.NTSEG1[7:0] + 1)$$
$$t_{BS2}= tq\ \cdot \  (FDCAN\_NBTP.NTSEG2[7:0] + 1)$$

> [!info] 
> Хотя NTSEG1 может быть от 1 до 256 tq а NTSEG2 может быть от 1 до 127 tq в классическом CAN BS1 от 1 до 16 tq а BS2 от 1 до 8 tq. Так что в классическом CAN bit time всего 25 tq. Однако stm32g4xx даёт куда большие настройки. 

> [!attention] 
> Настройка bit time производится только в Standby mode, т.е. регистры s FDCAN_DBTP и FDCAN_NBTP доступны только когда CCE и INIT в регистре FDCAN_CCCR равны 1. 

> [!attention] 
>  При использовании HAL вводится искусственное ограничение для NTSEG1 и NTSEG2. В структуре настройки для переменной fdcan из HAL, FDCAN_InitTypeDef, есть поле NominalTimeSeg1 и NominalTimeSeg2 которые по документации HAL должны быть от 2. То есть значение 1 которое возможно установить через CMSIS в  FDCAN_NBTP[NTSEG1] и FDCAN_NBTP[NTSEG2]. Я правда не знаю почему так сделано. В стандарте CAN от CiA написано что может быть от 1 tq что BS1 что BS2... Либо это прописали в ISO 11898-1: 2015 к которому у меня нет доступа. Либо это новодел от создателей HAL.

Для примера возьмём скорость 500 кбит/с. Тогда $bit\ time=\frac{1}{500000}=2$ мкс
Будем считать что у микроконтроллера частота 8 МГц, тогда
$t_{fdcan\_tq\_clk}=\frac{FDCAN\_NBTP.NBRP[8:0] + 1}{8000000}=125\text{ ns}$ при NBRP=0
Тогда чтобы получить 2 мкс нам нужно 2 мкс / 125 нс = 16 tq. Тогда нам в итоге нужно bit time = 16 tq чтобы у нас было 500 кбит/c

В [[CAN шина#Конфигурация]] было сказано что идеальным расположением sample point будет 87.5% от всего bit time. При этом нужно чтобы Phase_Seg2(это BS2 в FDCAN) должен быть больше чем время обработки информации контроллера CAN, которое, в зависимости от конкретной реализации, находится в диапазоне [0..2] tq, для stm32g4xx 0 tq 

Соответственно

$$(1+BS1)=0.875\cdot 16\text{ tq} = 14\text{ tq}$$
Тогда 
$$BS2=16-14=2 tq$$
SJW должен быть выбран как минимум из 4, Phase_Seg1 и Phase_Seg2. В нашем случае минимум это Phase_Seg2 = 2. Тогда SJW=2.

\le

	{
		2 \cdot (13 \cdot 16-2)

$$
$$II:\

$df \le 0.4\%$, что довольно хороший результат учитывая что максимальная толерантность 1.58%

# FDCAN для CAN-FD
Для работы с CAN FD нужно установить бит CCCR.FDOE. В случае CCCR.FDOE = 1, включена передача и прием CAN FD кадров.

В CAN пакете зарезервированный бит после идентификатора r0 в случае CAN-FD становится FDF битом: рецессивный FDF обозначает CAN FD кадр, а доминантный FDF обозначает классический CAN кадр. Значит передача CAN FD кадра или CAN кадра настраивается через бит FDF в соответствующем элементе буфера передачи.

Также появляются бит BRS который определяет будет ли изменяться скорость передачи данных внутри этого CAN FD пакета. Переключение скорости передачи данных CAN FD обозначается доминантным res и рецессивным BRS.

Больше про каждый бит в CAN кадре можно узнать в [[CAN-FD#Пакет CAN-FD]]

Когда FDCAN получает кадр с рецессивным FDF(что означает что пришёл CAN-FD кадр) но при этом вместо доминантного res пришёл рецессивный бит(что означает ошибку кадра) то FDCAN сигнализирует о событии исключения протокола, устанавливая бит PSR.PXE. Когда обработка исключений протокола включена (CCCR.PXHD = 0), это приводит к изменению состояния работы с приемника (PSR.ACT = 10) на синхронизацию с CAN шиной (PSR.ACT = 00) на следующем контрольном моменте. В случае, если обработка исключений протокола отключена (CCCR.PXHD = 1), FDCAN рассматривает рецессивный бит res как ошибку формата и отвечает кадром ошибки.

В FDCAN есть два режима для CAN-FD:
- Режим длинных кадров (LFM), где поле данных CAN-кадра может быть длиннее восьми байт(до 64 байт), включается при помощи CCCR.FDOE=1 и выставлением флага FDF в элементе буфера передачи
- Режим быстрых кадров (FFM), где поле управления, поле данных и поле CRC CAN-кадра передаются с более высокой скоростью передачи данных по сравнению с началом и концом кадра. Включается при помощи CCCR.BRSE=1 и выставлением флага BRS в элементе буфера передачи.

При CCCR.FDOE = 1 и CCCR.BRSE = 0, оценивается только бит FDF элемента буфера передачи. При CCCR.FDOE = 1 и CCCR.BRSE = 1, включена передача CAN FD кадров с переключением скорости передачи данных. Вариант с CCCR.FDOE = 0 и CCCR.BRSE = 1 невозможен.

> [!note] 
>  Изменение режима во время работы CAN рекомендуется только при следующих условиях:
> - Уровень ошибок в фазе данных CAN FD значительно выше, чем в фазе арбитража CAN FD. В этом случае отключите опцию переключения скорости передачи данных CAN FD для передач.
> - Во время запуска системы все узлы передают классические CAN сообщения, пока не будет подтверждено, что они могут общаться в формате CAN FD. Если это так, все узлы переключаются на работу в формате CAN FD.
> - Сообщения пробуждения в частичном сетевом взаимодействии CAN должны передаваться в классическом формате CAN.
> - Программирование в конце линии в случае, если не все узлы поддерживают CAN FD. Узлы, не поддерживающие CAN FD, удерживаются в тихом режиме до завершения программирования. Затем все узлы переключаются обратно на классическую связь CAN.

## Bit timing
При установленном бите CCCR.BRSE = 1 FDCAN может передавать быстрые кадры с изменённой скоростью для передачи "Data field" при установленном флаге BRS в элементе буфера передачи, так и принимать кадры с изменённой скоростью. 

![[Pasted image 20250605161740.png]]

То есть в CAN-FD arbitrage field и control field передаются с одной скоростью с одним bit time который определяется в FDCAN_NBTP(как и при настройке bit timing для CAN classic с той лишь разницей что CAN FD даёт делать bit time вплоть до 382 tq в отличии от 25 tq в CAN classic)

А data field и CRC field уже может работать куда быстрее(вплоть до 12 Мбит/с) и для этих полей используется свой bit time который настраивается через FDCAN_DBTP.
