# Base
SPI(Serial peripheral interface) - это интерфейс для работы с периферийными устройствами. SPI протокол поддерживает полудуплексный, полнодуплексный и симплексные режимы коммуницирования с периферийными устройствами. 

## Структурная схема SPI в STM32G4
![[Pasted image 20240217213050.png|800]]
*Диаграмма SPI*
Итак, `MOSI`(Master Out / Slave In data) используется если наш микроконтроллер является ведущим, а периферия ведомым, т.е. МЫ передаём В ПЕРИФЕРИЮ
`MISO`(Master In/ Slave Out data.) соответственно наоборот ПЕРИФЕРИЯ передаёт МЫ принимаем
`SCK`(Serial Clock) - это синхросигнал который синхронизирует как наш микроконтроллер, так и периферию.
`NSS` (Slave Select) - используется для выбора конкретного устройства (в режиме slave) для обмена данными. Когда NSS активен (обычно устанавливается в низкий уровень), устройство SPI в режиме master начинает обмен данными с выбранным slave-устройством. Когда NSS неактивен (обычно высокий уровень), slave-устройство игнорирует данные, отправленные по шине SPI. (ПРОСТЫМИ СЛОВАМИ он подаёт низкий уровень говоря slave устройству, что пора начинать обмен данными, соответственно даёт высокий уровень чтобы сказать что обмен - ВСЁ!). Управление NSS может быть программируемым и может быть настроено как аппаратный вывод GPIO, то есть на любую ножку можно поставить NSS.
`Shift Register` используется для хранения данных которые мы сейчас будет отправлять. То есть `shif register` собирает данные что мы хотим отправить и затем по битно сдвигом(по тактовым сигналам) отправляет данные через MOSI. Точно также `shif register` принимает данные через MISO. Упрощённо такая картина выглядит так:
![[Гиф 7.gif]]
А сам shift register выглядит примерно так:
![[Гиф 8.gif]]
*Да этот пример взят из работы UART не занудничай >:3*
`Tx FIFO`  сохраняет в себе значения для передачи и отправляет на shift register по принципу "первый пришёл первый вышел", а `Rx FIFO` соответственно сохраняет данные которые получаются из периферии в `Shif register`. То есть как бы `shift register` это последний рубеж, скорее сборщик данных перед отправкой. А `Tx FIFO` и `Rx FIFO` именно сохраняют значение которые мы передаём с контроллера. Т.е. когда мы что-то подаём на SPI->DR то мы подаём именно в конец Tx FIFO, соответственно если мы производим операцию чтения из SPI->DR мы читаем данные из начала Rx FIFO. Размеры `Tx FIFO` и `Rx FIFO` равны 32 бита
`CRC controller`(Cyclic Redundancy Check) контроллер используется для проверки целостности данных, передаваемых по цифровому интерфейсу. CRC контроллер вычисляет специальный код, называемый контрольной суммой, для блока данных, который будет передан по интерфейсу. Контрольная сумма вычисляется на основе полиномиального деления и представляет собой остаток от деления исходного блока данных на определенный полином.(короче типикл проверка она во всех интерфейсах есть).
`Baud rate generator` используется для работы с SCK он делит частоту тактирования для работы с периферией.
`Communication conttroller` ваще правит всем, он тут босс.

# Передача и приём данных
## RXFIFO и TXFIFO
Как я уже говорил когда мы передаём в SPI->DR данные на самом деле мы передаём данные в конец Tx FIFO, а когда мы берём значения из SPI->DR то на самом деле мы берём данные в начале Rx FIFO

Есть несколько флагов которые говорят нам о том что происходит в Rx FIFO и в Tx FIFO:
- `TXE` - показывает что Tx FIFO пуст
- `RXNE` - показывает что Rx FIFO не пуст
`RXNE` может вызываться при разных условиях которые определяется флагом `FRXTH`  в SPIx->CR2.
![[Pasted image 20241114230918.png]]
Т.е. если у нас установлен в 1 то RXNE будет ставиться в 1 если занято 8 бит Rx FIFO, в ином случае 16

Невероятно важно осознать то, что все данные для передачи через MOSI хранятся в TxFIFO, а не передаются напрямую через SPI_DR. Если этого не осознавать, то могут возникнуть ошибки. Распространённой ошибкой при работе с SPI является отправка большего количества битов чем мы ожидаем. Так SPI можно настроить на несколько размеров данных от 4 до 16 бит, настраивается это через биты DS[3:0] в регистре SPI_CR2. И соответственно если мы настроим на 8 битов, то мы будем хотеть передавать эти 8 битов записывая их в SPI_DR.

```c
SPI->DR = address; // address is uint8_t variable
```

И мы будем ожидать что мы будем передавать 8 битный address через SPI. Но если мы так сделаем то вместе с нужными нам 8 битами address мы отправим байт состоящий из 0.

![[Pasted image 20250820174405.png|150]]

Почему так происходит? Тут нужно вернуться к вопросу про TxFIFO и тому, что SPI_DR на самом деле является интерфейсом для работы с TxFIFO и RxFIFO

И сам SPI_DR является 16 битным
![[Pasted image 20250820174536.png]]
Поэтому компилятор в строке
```c
SPI->DR = address; // address is uint8_t variable
```

Делает неявное преобразования типа для address делая его 16 битным uint16_t заполняя старший бит нулями. И эти таким образом начиная со старшего бита это всё будет закидываться в TxFIFO по 8 байт(т.к. то каким образом данные закидываются в TxFIFO определяется битами DS[3:0]). И так как последний байт с нашим address заходит последним в TxFIFO то он первый и отправляется.

Соответственно чтобы всё работало так как мы ожидаем, мы отправляли на TxFIFO 8 бит нам нужно обращаться к адресу SPI_DR как к 8 битному указателю. То есть нам нужно использовать такую строку

```c
*((volatile uint8_t *) &SPI1->DR) = address;
```

## Как происходит передача/приём
В SPI как и I2C мастер передаёт тактирование слейву. Однако в отличии от I2C SPI линия даты и линия тактирования работает отдельно. Что я имею под этим ввиду. В SPI у нас есть всего 4 пина. Первый пин для передачи данных мастером, а соответственно и приёма слейвом(**MOSI** Master Out / Slave In). Второй пин для приёма мастером а значит и передачи слейвом(**MISO** Master In/ Slave Out). Третий пин собственно говоря для передачи тактирования для слейва(**SCK** Serial Clock). И четвёртый пин для выбора конкретного устройства которому наш мастер будет передавать данные(**NSS** Slave Select). Однако этот пин может быть простым gpio и для каждого слейва нужен свой отдельный NSS пин, но об этом мы поговорим отдельно в следующей подглаве.

Итак теперь самое главное как же у нас происходит передача данных в SPI. Смотри когда мы начинаем передачу, то мы передаём наши данные через пин MOSI, но вместе с этим мы начинаем передавать и тактирование. И теперь следи за руками, тактирование не идёт постоянно, оно идёт только в том случае если наш мастер передаёт какое-то сообщение, просто так тактирование идти не будет. Это делает важную особенность SPI.

Допустим у нас будет такая ситуация, нам нужно получить данные из какого-то регистра нашего периферийного устройства(т.е. слейва). Мы отправляем запрос на чтение через MOSI, затем отправляем адрес регистра из которого нам нужно прочитать данные через MOSI, а дальше что? Да в целом пока мы отправляем данные через MOSI слейв может в то же время передавать данные через MISO. Но то что он это МОЖЕТ делать, не значит что он это БУДЕТ делать, т.к. пока он не примет полностью адрес он не будет знать из чего собственно говоря брать информацию и будет находится так сказать в режиме приёма, а сам отправлять ничего не будет. Так а как прочитать то данные? Тактирования для слейва не будет, ничего через MISO идти не будет. БЕДА! Но она абсолютно не серьёзная. Ведь наша задача заключается в том, чтобы дать тактирование для нашего устройства, а значит нам нужно просто что-то передать через MOSI и сам SPI вместе с этой передачей будет давать слейву тактирование, что позволит слейву передать своё сообщение через MISO. И что мы будет отправлять через MOSI не имеет смысла, так как после того как мы отправили данные адреса регистра теперь слейв подготовил данные для передачи и сидит в режиме ожидания передачи. Поэтому если хотят уже конкретно принять данные со слейва просто передают по MOSI 0, тем самым как-бы начав передачу, а как бы и ничего и не отправив, но главное что мы дали тактирование слейву!

> [!tldr] 
> Если нам нужно прочитать данные то нам нужно отправить команду чтения через MOSI, затем адрес регистра из которого мы будем читать данные, точно также через MOSI, а после этого нам нужно снова отправить данные через MOSI но теперь просто 0, это нужно для того чтобы дать тактирование для слейва который уже отправит по MISO свои данные

## NSS
NSS говорит слейву о том что именно ЕМУ сейчас придёт сообщение, ну или то что он когда придёт тактирование должен передать сообщение. Короче говоря NSS говорит слейву что с ним сейчас происходит общение. Для начала общения NSS должен быть опущен в 0, а для конца общение должен быть поднят на 1. Когда NSS будет поднят в 1, чтобы ты там не писал через MOSI он всё пропустит.

## Работа с 2мя и более слейвами
Как я и говорил NSS используется для выбора слейва. Опять же как я и говорил выше если NSS будет поднят в 1 то тогда слейв будет полностью игнорировать сообщения что идут с SPI. Поэтому если подключить кучу слейвом к одному SPI при помощи отдельных NSS можно управлять с кем будет происходить общение.

## Варианты работы с NSS
Есть несколько вариантов как можно работать с NSS. 
- **Программное управление NSS (SSM = 1)**: В этой конфигурации информация о значении NSS для самого МК управляется внутренне значением бита SSI в регистре SPIx_CR1. Но вот внешний контакт NSS свободен для других применений. То есть так мы отключаем зависимость логики SPI от внешнего контакта NSS.
	Сам микроконтроллер тоже является SPI устройством соответственно он тоже должен следить за значением NSS, если с ним начинают общение. Но в режиме мастера нам это не к чему, ведь мастер сам начинает общение со слейвом, он сам управляет пином NSS, а не следит сам за тем, какой на данный момент NSS. Соответственно SSM=1 настройка нужна для мастер мода

В этом случае если у нас SSI = 1, то микроконтроллер внутренне считает что NSS постоянно активным т.е. находится в 0. Поэтому если слейв будет отправлять ему данные микроконтроллер будет их всегда получать
Если SSI=0, то микроконтроллер внутренне считает что CS постоянно неактивным т.е. находится в 1.
Но вообще когда у нас SSM=1, то конкретно при работе со слейвом, мы работаем с самым обычным gpio как с NSS. Т.е. мы подключаем обычный gpio к NSS на слейве, чтобы он понимал началась ли передача. Но SSI управляет тем как сам микроконтроллер считает, есть ли передача или нет.

Теперь если у нас апаратное управление NSS
**Режим SSOE**
-  **Включение выхода NSS (SSM=0, SSOE=1)**: Эта конфигурация используется только тогда, когда микроконтроллер настроен как ведущий. Контакт NSS управляется аппаратно. Сигнал NSS устанавливается в низкий уровень, как только SPI включен в режиме ведущего (SPE=1), и остается в низком уровне до тех пор, пока SPI не будет отключен (SPE=0).
- **Отключение выхода NSS (SSM=0, SSOE=0)**: Если микроконтроллер действует как ведущий на шине, эта конфигурация позволяет использовать несколько ведущих. Если контакт NSS установлен в низкий уровень в этом режиме, SPI переходит в состояние ошибки ведущего и устройство автоматически перенастраивается в режим ведомого. В режиме ведомого контакт NSS работает как стандартный вход "выбор микросхемы", и ведомое устройство выбирается, пока линия NSS находится на низком уровне.

**Режим NSSP**
В режиме NSSP у нас NSS будет управляться самим микроконтроллером. То есть он будет переходить в 0 когда у нас будет происходить общение, и сам переходить в 1 когда общение закончилось. В этом режиме нужно убрать флаг CPHA=0, иначе NSSP не будет работать.
У этого режима есть одна проблема. Если ты настроишь DMA для SPI и будешь отправлять кучу сообщений последовательно, то NSSP будет силой поднимать CS в 1 на 1 такт SPI_SCK, поэтому у нас будут небольшие перерывы в отправке.

## Как устройства сохраняют данные в перерывах передач
У тебя может возникнуть вопрос, вот я отправил первое сообщение где дал адресс, затем принял первую часть данных, а вот вторая осталась ещё в устройстве с которого мы снимаем значения. Что же с ними будет?
Дело в том что устройство держит данные до того пока не закончиться с ним общение. То есть если он должен отправить 16 бит, но пока отправил 8, то он будет держать 8 бит пока мы не подадим тактирование.
А общение заканчивается когда NSS переходит в 1. Вот такие дела.

По этой же причине режим NSSP довольно странный потому что у нас поднимается постоянно NSS в 1. Что сбрасывает данные на устройстве, тем самым ломая передачу.

# Настройка SPI
![[Pasted image 20240217225110.png]]
На русском:
Процедура конфигурации практически одинакова для мастера и ведомого. Для специфичных настроек режима следуйте соответствующим разделам. Когда необходимо инициализировать стандартное соединение, выполните следующие шаги:

1. Запишите соответствующие регистры GPIO: настройте GPIO для пин MOSI, MISO и SCK.
2. Запишите в регистр SPI_CR1: 
- a) Настройте скорость тактового сигнала последовательного интерфейса с помощью битов BR[2:0] (Примечание: 4). 
- b) Настройте биты CPOL и CPHA для определения одного из четырех отношений между передачей данных и тактовым сигналом (бит CPHA должен быть сброшен в режиме NSSP) (Примечание: 2 - за исключением случая, когда CRC включен в режиме TI). 
- c) Выберите простой дуплекс или полудуплекс, настроив RXONLY или BIDIMODE и BIDIOE (RXONLY и BIDIMODE не могут быть установлены одновременно). 
- d) Настройте бит LSBFIRST для определения формата фрейма (Примечание: 2). 
- e) Настройте биты CRCL и CRCEN, если необходим CRC (при этом тактовый сигнал SCK должен находиться в неактивном состоянии). 
- f) Настройте биты SSM и SSI (Примечания: 2 & 3). 
- g) Настройте бит MSTR (в многомастерной конфигурации NSS избегайте конфликтного состояния на NSS, если мастер настроен для предотвращения ошибки MODF).
3. Запишите в регистр SPI_CR2: 
- a) Настройте биты DS[3:0] для выбора длины данных для передачи. 
- b) Настройте бит SSOE (Примечания: 1 & 2 & 3). 
- c) Установите бит FRF, если требуется протокол TI (сохраняйте бит NSSP сброшенным в режиме TI). 
- d) Установите бит NSSP, если требуется импульс NSS между двумя единицами данных (сохраняйте биты CHPA и TI сброшенными в режиме NSSP). 
- e) Настройте бит FRXTH. Порог RXFIFO должен соответствовать размеру доступа для чтения регистра SPIx_DR. 
- f) Инициализируйте биты LDMA_TX и LDMA_RX, если DMA используется в упакованном режиме.
4. Запишите в регистр SPI_CRCPR: настройте полином CRC, если это необходимо.
5. Запишите соответствующие регистры DMA: настройте потоки DMA, предназначенные для SPI Tx и Rx в регистрах DMA, если используются потоки DMA.

Примечания:
1. Шаг не требуется в режиме ведомого.
2. Шаг не требуется в режиме TI.
3. Шаг не требуется в режиме NSSP.
4. Шаг не требуется в режиме ведомого, за исключением ведомого, работающего в режиме TI.

## Пример
```c
/* какие ножки
     * PA5 <-> SPI_SCK
     * PA6 <-> SPI_MISO
     * PA7 <-> SPI_MOSI
     * PA4 <-> SPI_CS
*/

void spi_configure(void){
	// включаем тактирование для SPI1
	RCC->APB2ENR |= RCC_APB2ENR_SPI1EN;
	
	//делаем CPOL = 1 и CPHA = 1 смотри выше как будет выглядить тактирование
	SPI1->CR1 &= ~(SPI_CR1_CPOL_Msk | SPI_CR1_CPHA_Msk);
	SPI1->CR1 |= SPI_CR1_CPOL | SPI_CR1_CPHA;

	// выбираем master mode потому что наш микроконтроллер будет мастером
	SPI1->CR1 |= SPI_CR1_MSTR;

	// настраивам деление частоты на 2 смотри ниже расстановки BR
	// LSBFIRST отвечает будет ли little-indian или big-indian
	// у нас big indian то есть первым идёт больший бит(слева-направо)
	SPI1->CR1 &= ~(SPI_CR1_BR_Msk | SPI_CR1_LSBFIRST_Msk);

	// включаем управление слейвами софтвейрно SSM=1, то бишь мы вместо того чтобы
	// давать самому интерфейсу опускать пин NSS(для этого нужно использовать 
	// определённый пин настроенный на альтернативную функцию) мы можем использовать
	// любой пин в режиме вывода который мы должны ручками опускать чтобы начать 
	// пользоваться SPI и поднимать при конце общения через SPI
	// SSI позволяет работать через SSI с вводом выводом(не пользовался),  
	SPI1->CR1 |= SPI_CR1_SSM | SPI_CR1_SSI;
	// убераем только чтение, так как нам и запись нужна
	SPI1->CR1 &= ~(SPI_CR1_RXONLY_Msk);

	// сбрасываем конфигурацию CR2
	SPI1->CR2 = 0;

	// хз чё тут делает, он работает с FIFO но я его не включал
	SPI1->CR2 |= SPI_CR2_FRXTH;
	// выставляем размер данных в 8 бит, смотри ниже расценку
	SPI1->CR2 |= 0b0111 << SPI_CR2_DS_Pos;

	// собственно говоря включет SPI
	SPI1->CR1 |= SPI_CR1_SPE;
}

void gpio_configure(void){
	// включаем тактирование для gpioa
	RCC->AHB2ENR |= RCC_AHB2ENR_GPIOAEN;

	// настраиваем ножки 4 в вывод, 5-7 в альтернативную функцию
	GPIOA->MODER &= ~(GPIO_MODER_MODE4_Msk | GPIO_MODER_MODE5_Msk 
	| GPIO_MODER_MODE6_Msk | GPIO_MODER_MODE7_Msk);
	GPIOA->MODER |= 1 << GPIO_MODER_MODE4_Pos | 2 << GPIO_MODER_MODE5_Pos 
	| 2 << GPIO_MODER_MODE6_Pos | 2 << GPIO_MODER_MODE7_Pos;

	// делаем максимальную скорость ножек
	GPIOA->OSPEEDR |= (3 << GPIO_OSPEEDR_OSPEED4_Pos) | 3 << GPIO_OSPEEDR_OSPEED5_Pos 
	| 3 << GPIO_OSPEEDR_OSPEED6_Pos | 3 << GPIO_OSPEEDR_OSPEED7_Pos;

	// ставим 5 функцию для ножек 5-7, что нужно как раз для SPI1, смотри ниже расценку
	GPIOA->AFR[0] |= (5 << GPIO_AFRL_AFSEL5_Pos) | (5 << GPIO_AFRL_AFSEL6_Pos) 
	| (5 << GPIO_AFRL_AFSEL7_Pos);
}

// опускает значение на 4 пине, что отвечает за NSS
void cs_enable(void){
    GPIOA->BSRR = GPIO_BSRR_BS4;
}

// поднимает значение на 4 пине, что отвечает за NSS
void cs_disable(void){
    GPIOA->BSRR = GPIO_BSRR_BR4;
}
```

```c
uint8_t spi_transmit(uint8_t address){
    // ждём пока буфер передачи не будет пустым
    while((SPI1->SR & SPI_SR_TXE) != SPI_SR_TXE){}

	// ВАЖНО!!! этот код адрессуется к SPI1->DR в 8 битном формате напрямую потому что
	// в случае используемой переферии icm20600 у нас используются именно 8 бит памяти
    // если не адрессоваться к SPI1->DR как 8 битному напрямую то работать не будет
    *((volatile uint8_t *) &SPI1->DR) = address;

    // ждём пока ответ не появится в SPI1->DR
    while((SPI1->SR & SPI_SR_RXNE) != SPI_SR_RXNE){}

    return (SPI1->DR);
}
```

```c
void spi_read_reg(uint8_t addr, uint8_t *data){
	// включаем cs
	cs_enable();
	
	// отправляем адрес с поднятым большим битом, так как это надо
	// чтобы сказать устройству что мы хотим читать данные
	spi_transmit(addr|0x80);

	// отправляем мусорный адрес GARBAGE = 0, чтобы устройство
	// передало нам данные
	*data = spi_transmit(GARBAGE);

	// выключаем cs
	cs_disable();
}

void spi_write_reg(uint8_t addr, uint8_t data){
	// включаем cs
	cs_enable();

	// передаём для начала адрес без поднятого большего бита,
	// это значит мы хотим записать в этот адресс
	spi_transmit(addr);
	
	// передаём данные которые мы хотим передать в адресс
	spi_transmit(data);

	// выключаем cs
	cs_disable();
}
```

# Настройка DMA
## Почему это обязательно нужно при работе с spi
Дело в том что когда мы работаем с spi без dma при этом хотим отправить несколько последовательных сообщений, то spi будет делать довольно продолжительные паузы. А это пустая трата времени!
![[изображение 1.png]]
DMA же решает эту проблему делая последовательные сообщения действительно последовательными без бесполезных пауз
![[изображение 2.png]]
По сему непросто нужно использовать DMA при работе с SPI а при возможности НЕОБХОДИМО если у нас не занят DMA чем то другим.
