От самого гугла довольно непонятная документация но у nanopb есть своя документация где рассказывается и про то как работать c protobuf. Смысл этого protobuf я так понял в том, чтобы написать в каком-то стандартизированном независимом от языка формате то какие данные могут передаваться через какое-то средство связи. И соответственно на двух сторонах ты можешь encode или decode данные в эти структуры. У protobuf есть свой генератор protoc но у него нет варианта для генерирования Си кода, да и как я понял там получается довольно громоздкий код, поэтому для использования protobuf в Си и сделан nanopb.

Для того чтобы сгенерировать файл .proto нужно использовать nanopb_generator.py. Его можно скачать через pip
```
pip install nanopb
```

Единственная проблема в том, что если использовать options для nanopb, которые довольно полезны потому что например в protobuf нет uint16 uint8, только uint32. Я так понял у protobuf там какая-то своя умная система сжатия данных, что даже если мы выставили uint32 и отправляется например число 5, то во время кодирования оно сожмётся до 8 бит и только эти 8 бит, а не 32 бита отправятся. Но чтобы в Си в структурах использовался именно uint8_t переменные нужно использовать options которые определены в nanopb.proto, о options доступных в nanopb.proto написано в документации nanopb в API reference/Generator options

Единственная проблема, если мы будем использовать nanopb options то в таком случае нам понадобится путь к папке где будет располагаться nanopb.proto файл. Поэтому нужно либо во всех проекта тащить этот nanopb.proto файл дополнительно, либо прописывать директорию к нему.

Также если использовать питон, то он попросит также .py файл самого nanopb.proto.

В общем у метода довольно много подводных камней. Но как я вижу использование этого всего. Мол можно будет написать файл с расширением .proto где можно расписать сообщения с разными переменными и соответственно их можно будет использовать как в пайтоне, так и в си++ и т.д. При этом можно будет сделать дополнительный питон скрипт который на основе полученного .h файла со всеми структурами будет создавать `_instance.c` где создаст для каждого из сообщения структуру где будут храниться переменные. И соответственно при приёме данных мы будем проверять какое из сообщений пришло и передавать данные в соответствующие instance.

##  Использование Bit Field структуры

```c
typedef struct {
  uint8_t *buffer;              // buffer pointer
  uint16_t depth;               // max items

// =============== LOOK HERE ===============
  struct TU_ATTR_PACKED {
    uint16_t item_size    : 15; // size of each item
    bool     overwritable : 1;  // ovwerwritable when full
  };
// =============== LOOK HERE ===============

  volatile uint16_t wr_idx;     // write index
  volatile uint16_t rd_idx;     // read index
} tu_fifo_t;
```

При помощи Bit Field можно таким образом упаковывать какой нибудь флаг, заместо старшего бита другого поля.

> ```asm

> ``` 


## Bit banding и проверка на быстроту

В Cortex-M4 есть такая штука как bit banding которая позволяет работать с каждым битом в каком то адресе как с целым словом. В задумке это убирает проблему с изменением битов через `|=` и `&=` потому что они занимают целых 3 операции
```c
uint8_t temp = 0;
temp |= 0x4;
temp &= ~0x4;
```
И такой код будет переводится в такой ассемблер:
```asm
#temp |= 0x4;
ldrb r3, [r7, #7]
orr.w r3, r3, #4
strb r3, [r7, #7]

#temp &= ~0x4;
ldrb r3, [r7, #7]
bic.w r3, r3, #4
strb r3, [r7, #7]
```

Из-за того что каждая такая операция занимает 3 ассемблерные операции, это может оказаться проблемой при многопоточном программировании, или даже при прерываниях. Ведь у нас могут измениться в другом потоке или в обработчике прерываний эти биты ДО ТОГО как мы их изменим в основном коде.

А bit banding отображает КАЖДЫЙ БИТ определённой области памяти на целое слово. Пример такого отображения для адреса SRAM 0x20000000 приведён на рисунке
![[Pasted image 20250922213403.png]]
То есть это даёт нам обращаться к каждому биту как к отдельному слову, что на бумаге должно дать нам только 1 операцию присваивания.

Однако проблема в том, что я заметил, что если у нас слишком большой 32 битный адрес, как в случае с 0x22000000 gcc сделает операции загрузки сначала этого адреса из конца области функции, а затем только будет работать с данными в этом адресе.

Ну то есть он будет делать также и при работе с `|=` и `&=` и как бы bit banding действительно избавит от нескольких лишних тактов и уберёт проблему многопоточности. Однако bit banding не позволяет записывать одновременно несколько битов, что очень полезно при настройке. И например в USB когда нам нужна быстрая настройка во время Enumeration, мы будем отдавать предпочтения `|=` и `&=` поэтому я бы протестировал то как это всё будет работать в дизассемблированном тексте.

Про bit banding можно почитать в любом RM на stm32 с Cortex-M4 процессором. У него обычно есть часть в главе про память stm32. Или просто поиском ищи `bit banding`
