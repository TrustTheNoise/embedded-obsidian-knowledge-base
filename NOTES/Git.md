# База
## Базовые команды
### git init
- Создает скрытую директорию `.git` в текущем каталоге.
- В этой директории хранятся все данные и конфигурации репозитория.

По факту же это и является самым начальным отсчётом для гита в твоём проекте. git init создаёт локальный репозиторий твоего проекта. Только после git init ты можешь писать остальные команды

### git status
Представим: ты добавил какие то файлы в свой проект. Только вот то что их добавил на своём компьютере не значит что гит эти данные зафиксировал. Чтобы узнать какие изменения в твоём проекте гит зафиксировал а какие нет используется git status

### git add
Когда ты добавишь какие то файлы в твой репозиторий то тебе нужно их собственно говоря отдать самому гиту. Для этого и используется git add
### git commit
Commit можно упрощённо представлять как скриншот того как выглядел твой проект в определённое время. То есть вот ты зафиксирвал для гита изменения через git add теперь ты хочешь чтобы гит запомнил такие изменения если вдруг ты впоследствии захочешь вернуться к ним, для этого и используется git commit
```sh
git commit -m "Commit message"
```

### git log
Показывает историю твоих коммитов
- Отображает список всех коммитов в репозитории.
- Показывает хэш коммита, автора, дату и сообщение коммита.

### git remote add
**Описание**: Добавляет удаленный репозиторий.
```sh
git remote add <name> <url>
```
- `<name>`: Имя для удаленного репозитория (обычно `origin`).
- `<url>`: URL удаленного репозитория.
- Добавляет удаленный репозиторий, к которому можно будет пушить и пуллить изменения.

Удалённый репозиторий это как раз и есть репозиторий на гитхабе, то есть url должен быть из репозитория на гитхабе
### git push
Отправляет изменения в удаленный репозиторий.
```sh
git push <remote> <branch>
```
или
```sh
git push -u <remote> <branch>
```

- `<remote>`: Имя удаленного репозитория (обычно `origin`).
- `<branch>`: Имя ветки, которую вы хотите запушить (обычно `main` или `master`).
- `-u`: Устанавливает удаленную ветку как отслеживаемую для текущей ветки.
-u как бы синхронизирует локальный и удалённый репозиторий тем самым мы теперь можем писать просто git push 
### git pull
Получает изменения из удаленного репозитория и объединяет их с локальными.
```sh
git pull <remote> <branch>
```
- `<remote>`: Имя удаленного репозитория (обычно `origin`).
- `<branch>`: Имя ветки, из которой вы хотите получить изменения (обычно `main` или `master`).
- Объединяет изменения из удаленного репозитория с локальными изменениями.
  
  Опять же если мы один раз используем флаг -u то больше не нужно будет вводить имя удалённого репозитория, а просто git pull
### git clone
Клонирует удаленный репозиторий на локальную машину.
```sh
git clone <url>
```
- `<url>`: URL удаленного репозитория.
- Создает локальную копию удаленного репозитория.
То есть если ты хочешь скачать чужой репозиторий на гитхабе себе, то как раз и используй git clone

## Пример: push данных в гитхаб
```sh
# Создайте новую директорию и перейдите в нее
mkdir my_project
cd my_project

# Инициализируйте новый репозиторий
git init

# Создайте файл и добавьте его в репозиторий
echo "Hello, World!" > README.md

# Добавьте файл в индекс
git add README.md

# Сделайте коммит
git commit -m "Initial commit"

# Проверьте состояние репозитория
git status

# Просмотрите историю коммитов
git log

# Создайте новый репозиторий на GitHub и скопируйте его URL

# Добавьте удаленный репозиторий
git remote add origin https://github.com/your_username/my_project.git

# Запушьте изменения в удаленный репозиторий
git push -u origin main
```

# Ветки и merge
**Создание новой ветки**:

- `git branch <branch_name>`

**Переключение на другую ветку**:

- `git checkout <branch_name>`

**Создание и переключение на новую ветку одновременно**:

- `git checkout -b <branch_name>`

**Просмотр всех веток**:

- `git branch`

**Слияние ветки**:
- `git merge <branch_name>`

**Удаление ветки**:
- `git branch -d <branch_name>`

## Пример работы с ветками

```sh
# Создайте новую ветку и переключитесь на нее
git checkout -b feature-branch

# Внесите изменения и сделайте коммит
echo "New feature" > feature.txt
git add feature.txt
git commit -m "Add new feature"
```

```sh
# Переключитесь обратно на основную ветку
git checkout main
```

```sh
# Объедините изменения из feature-branch в основную ветку
git merge feature-branch
```

## Справляемся с error merge
Если в ветках которые мы merge будет в одинаковых строках разные данные то гитхаб не будет понимать что с этим всем делать, поэтому он предоставляет пользователю разобраться что ему нужно из обоих ветках

Как это происходит
```sh
git merge feature-branch
```
1. **Если возникает конфликт**, Git сообщит вам об этом и пометит конфликтующие файлы. Откройте эти файлы и найдите конфликтующие блоки, которые будут выглядеть примерно так:
```plaintext
<<<<<<< HEAD
Your changes
=======
Changes from feature-branch
>>>>>>> feature-branch
```
2. Вручную отредактируйте файлы, чтобы объединить изменения. И удалите маркеры конфликтов (`<<<<<<<`, `=======`, `>>>>>>>`).
3. Добавьте изменённые файлы в гит
```sh
git add <file>
```
4. Завершите слияние
```sh
git commit -m "Resolve merge conflict"
```

# Возвращение по коммитам
Я рассказал про git checkout ещё в ветках, но git checkout может быть полезен для возвращения к прошлым коммитам в одной ветке, для этого нужно написать вместе с git checkout хэш самого коммита т.е.
```sh
git checkout <hash_of_commit>
```
НО! Такое нужно только для просмотра что было в прошлых коммитах. Мы не удаляем остальные коммиты которые были после этого коммита, мы даже таким образом не можем ничего менять в этом коммите, гит даст ошибку, это просто просмотр что было в том изменении.
Но что если мы хотим всё же вернуться к прошлому коммиту, допустим у нас есть баг в последующий коммитах поэтому мы хотим вернуться к прошлому коммиту чтобы всё исправить.
Для такого есть несколько путей
## git reset
Git reset возвращает нас на прошлый коммит и удаляет все последующие после этого коммиты. Но при этом мы можем выбрать что же мы хотим оставить из удалённых коммитов
### soft reset
```sh
git reset --soft <hash_of_commit>
```
Возвращает к коммиту с `<hash_of_commit>` и все коммиты после него передаёт изменения из удалённых коммитов в git add тем самым мы как бы сразу принимаем все изменения из остальных коммитов, осталось только написать git commit и мы как бы удаляем все прошлые коммиты но при этом создаём 1 новый коммит со всеми изменениями из удалённых коммитов.
Но мы всё равно можем убрать какие-то изменения и только потом коммитнуть
### mixed reset
```sh
git reset <hash_of_commit>
```
Вся разница от soft reset в том что мы не добавляй изменения из удалённых коммитов в git add поэтому мы изначально планируем что будем что-то менять
### hard reset
```sh
git reset --hard <hash_of_commit>
```
Hard reset просто удаляет все коммиты после нужного нам коммита, без шанса на возвращение фичей от туда >:3

## git revert
Теперь представим такую ситуацию, ты опять же понаделывал ошибок в коммитах и хочешь вернуться назад. Однако удалять коммиты всё таки не комильфо, тим лид же должен знать какой ты долбаёб. А что если просто возвращаться к прошлому коммиту, поубирать все баги и просто добавить как новый коммит, не удаляя вообще ни одного коммита!
Для этого и нужен get revert
```sh
git revert <hash_of_commit>
```
git revert схож по работе с git merge но только с разными коммитами. То есть он как бы объединяет(merge) коммит с `<hash_of_commit>`  со всеми последующими коммитами. И да, если опять же на одинаковых строках у этих коммитов будут разные данные то у нас будет revert error и мы точно также как и merge error должны ручками всё это исправлять.
Ну а затем уже коммитить этот git revert. Профит!

# Сохранение работы на потом или git stash
Теперь представим другую ситуацию. Мы сидим делаем какую то фичу, никого не трогаем. И тут тим лид врывается в чат и просит тебя исправить баги, и это нужно было ещё вчера потому что нужно показывать проделанную работу кабан-кабанычу. Получается не закомитишь неготовую фичу но при этом удалять тоже не хочется, потом же опять всё это по новой начинать. И что же делать. У гитхаба есть дополнительный карман в куртке где он может хранить твой неготовый для коммита код и при этом ты сможешь поисправлять баги для тим лида и затем доделать свой код. Для этого и используется git stash
```sh
git stash
```
После того как ты это напишешь код что ты писал после последнего коммита пропадёт и ты сможешь спокойно исправлять баги, а затем коммитнуть эти исправления. После этого тебе собственно говоря нужно вернуть из git stash твой незаконченный код. Но как? Для начала нам нужно узнать название нашего stash что мы сделали для этого мы пишем
```sh
git stash list\
```
И мы получим что-то типо такого
![[Pasted image 20241027155335.png]]
Теперь мы видим что название нашего stash это `stash@{0}`. Теперь мы его пишем в комманду
```sh
git stash apply <stash_name>
```
И у нас опять же работает такая же тема как с merge error
