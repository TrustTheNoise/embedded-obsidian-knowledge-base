# Для чего нужны ld файлы

Как мы разобрались в [[Компилятор и линковщик#Запуск исполняемого файл]] в коде запускаемом на ОС адрес для самого кода(т.е. для сегмента .text) выбирается в одном адресе, а уже ОС закидывает эти данные куда ему это удобно, чтобы собственно говоря этот код исполнять.

В микроконтроллерах же так не получится, т.к. там нет ОС который сможет решить куда закидывать код, у нас есть только bootloader который просто закидывает данные на flash память, а нам уже нужно через код в си, правильно закидывать данные в нужные участки памяти разные секции, такие как .bss и т.д. А также мы должны правильно выставить адреса для стека и кучи. Ведь всё это тоже делает ОС и стандартная библиотека. А в случае микроконтроллеров, это всё переходит на руки программиста.

Поэтому нам важно объяснить какие участки памяти для чего используются в системе на которую мы компилируем наш файл. Раз исполняемый файл создаются линковщиком, поэтому нам нужно для этого линковщика создать объяснение об адресации, для этого и используется linker scripts. Плюс некоторые из переменных для адресаций что мы будем расписывать в linker script будут видны также и в си коде, что будет довольно удобно. Какие переменные из linker script доступны также в си коде будет рассказано в разделах ниже.

Для GNU ld есть стандартный linker script который описан для каждой ОС. То же самое есть и для arm-none-eabi-ld, однако стандартный linker script не подходит для каждого микроконтроллера, ведь у каждого микроконтроллера на arm различается расположение сегментов памяти на адресное пространство. Поэтому нам нужно самому его создавать. 

Для того чтобы понимать каким образом создавать ld файл для какого-либо микроконтроллера и написана эта заметка.

# ENTRY команда 

ENTRY команда выставляет адрес с которого должна начинаться программа, обычно туда выставляется имя функции с которой должна начинаться программа, то есть мы выставляем допустим какую-то функцию из Си или ассемблера в линкер скрипте в эту команду.
```ld
ENTRY(_symbol_name_)
```
Например в stm32 стартовой точкой должна быть функция адрес которой помещена на 2 элемент в vector table. Можно просмотреть допустим что туда помещает сгенерированный cubeide startup_stm32g474.s. 
```asm title=startup_stm32g474.s
  .section .isr_vector,"a",%progbits
  .type g_pfnVectors, %object
  .size g_pfnVectors, .-g_pfnVectors

g_pfnVectors:
  .word _estack
  .word Reset_Handler
  .word NMI_Handler
  .word HardFault_Handler
  ...
```
Как можно заметить вторым элементом стоит адрес на функцию Reset_Handler. Поэтому для этого микроконтроллера выставляется такая ENTRY комманда
```ld
ENTRY(Reset_Handler)
```

Но ведь stm32 на хардверном уровне будет считать Reset_Handler стартовой точкой, для чего нам ещё это писать софтверно в ld файле? 
Дело в том, что нам может понадобится дебажить наш elf файл. И GDB сам не знает с какой точки начинается исполнение elf файла, поэтому он и будет подсматривать в ENTRY команду чтобы узнать с чего начинается исполнение 

# MEMORY команда
Этот блок определяет доступные регионы памяти в системе такие как FLASH, RAM, указывая их стартовые адреса(ORIGIN), размеры(LENGTH как в B, так и в K(килобайт) или например M(мегабайт)) и права доступа(attr).

эта команда описывается таким образом
```ld
MEMORY
{
    name (attr) : ORIGIN = 0x1488228, LENGTH = 1337KB
}
```

Все название для регионов памяти не имеют никакой разницы вне linker script. Поэтому его можно назвать как угодно, как удобно. ВВсе данные что будут расписаны в этой команде будут использоваться при расставлении линкером секций в [[#SECTION команда]]. Соответственно также эти данные помогают понять сколько памяти было занято, а также если какая-то секция в итоге заняла больше данных чем мы выставили в MEMORY команде, то линкер выдаст ошибку.

Имена регионов хранятся в отдельном пространстве имён и не будут конфликтовать с именами символов, файлов или секций. Каждый регион памяти должен иметь уникальное имя в рамках команды **MEMORY**.

Права доступа обозначаются комбинацией символов: 
- R (Read-only) - Размещает раздел памяти как **только для чтения**.
- W (Read/Write section) - Разрешает **чтение и запись** в секцию.
- X(Executable section) - Помечает регион как **исполняемый**. Это значит, что процессор может выполнять код из этого региона.

# SECTION команда
Этот блок определяет размещение различных секций в регионах памяти, описанных в блоке **MEMORY**. Секции вроде **.text**, **.data**, **.bss** описываются в этом блоке. Этот блок определяет куда линкер будет закидывать все секции из объектных файлов при создании исполняемого файла. Мол в каждом объектном файле создаются свои секции для .text для .bss и так далее. Линкер же их должен закинуть в одни общие секции для .text для .bss и так далее.

Но линкеру нужно знать по какому адресу ему нужно всё это в конечном итоге закидывать. По каким адресам должны располагаться конечные секции. Поэтому мы и указываем в какие разделы памяти закидывать каждую секцию, и в каком порядке в памяти будут расположены эти секции. То есть порядок секций в SECTION команде имеет значение
Синтаксис для SECTION таков:
```ld
SECTIONS
{
    .text:
    {
        //body of section
    }>(vma) AT> (lma)
}
```

- **VMA (Virtual Memory Address)** — адрес, по которому **секции будут находиться и использоваться во время выполнения программы**. То есть откуда будет выполняться код или читаться/писаться данные.
- **LMA (Load Memory Address)** — адрес, **откуда они будут загружены при старте** (в прошивке/флеше).

Для примера возьмём секцию .data в которой хранятся инициализированные глобальные переменные. Изначально данные  для этих переменных находятся во флеше, однако при выполнении программы мы должны брать эти глобальные инициализированные данные из RAM. Таким образом нам нужно для линкера указать, что сначала данные из этой секции находятся в FLASH, а затем переносятся в RAM, при помощи синтаксиса

```ld
SECTIONS
{
    .data:
    {
        //body of section
    }>RAM AT >FLASH
}
```

Знание линкера где будет находится секция при загрузке прошивки, т.е. LMA, нужно для того чтобы линкер правильно закинул в elf файле эту секцию. А знание где эта секция будет при выполнении программы, нужно для того чтобы уже в коде Си, когда бы мы обращались к глобальным переменным, линкер правильно изменял адрес на секцию из vma, а не lma

Если же у нас при запуске программы и при выполнении данные находятся в одинаковом месте, то можно писать без AT >(lma)

А что же должно находится в теле для каждой секции. Мы должны сказать что линкер должен помещать в каждую из этих секций, например в .text он должен помещать секции всех объектных файлов. Можно руками написать .text каких файлов нам нужен при помощи
```ld
.text:
{
    .text of main.o some_file.o startup.o
}
```
А можно написать при помощи шаблона `*(section)` где `*` которая "все входные объекты". В итоге это будет выглядеть вот так:
```ld
.text:
{
    *(.text)
    foo.o(.text_foo)
}> FLASH
```
И таким образом линкер будет закидывать в итоговую секцию .text, которая будет находится во FLASH памяти, все .text секции из всех входных объектных файлов. А также из объектного файла foo.o он закинет секцию .text_foo

Тут работает как и со всеми шаблонами, мол можно написать `*(.text*)` и в таком случае будет помещаться все секции что подходят под шаблон `.text*` т.е. .text_rel .text_rel_o и тому подобное

## Location counter and script symbols
Часто нам нужно в си коде узнать где находятся границы каких-либо секции. Для примера возьмём ту же .data секцию. Нам нужно перенести данные с FLASH в RAM. Ведь мы должны это сделать сами через Startup код на си. Соответственно нам нужно знать адреса начала для RAM и для FLASH секции .data

Для этого нам и нужны script symbols. Эти символы хранят адреса из линкер скрипта, которые в дальнейшем можно будет использовать в си коде.

> [!important] 
> Символы linker script это не переменные, это именно адреса. То есть если мы напишем в linker script
> ```ld
> start_of_flash = 0x8000000
> ```
>  То мы не получим переменную со значением 0x8000000, мы получим переменную НАХОДЯЩУЮСЯ ПО АДРЕСУ 0x8000000, которую уже можно будет использовать в си используя
>  ```c
>  extern start_of_flash
>  ```
>  И тогда адрес этой переменной будет `&start_of_flash = 0x8000000`

Теперь нам нужно знать где находится начало .data сегмента во FLASH и в RAM. Для того чтобы узнать адрес в текущем положении в linker script можно воспользоваться символом `.` называемый linker counter который даёт текущий адрес, т.е. в скрипте
```ld
SECTION 
{
    .data :
    {
        _sdata = .
        *(.data)
        _edata = .
    }> RAM AT >FLASH
}
```
Мы создали символы для начала .data секции и конца .data секции <span style="color:red">В РАЗДЕЛЕ ПАМЯТИ RAM!</span> После того как линкер закинет в эту секцию все секции .data из всех объектных файлов, мы получим правильные значение для двух `.` в начале и в конце. Т.к. символ `.` (location counter) двигается по мере того, как линкер размещает содержимое (`*(.data)` и т.д.), поэтому значения `_sdata` и `_edata` автоматически получаются правильными.

Поэтому порядок расположения секций в SECTION так важен, т.к. линкер будет именно в таком порядке расставлять их в разделах памяти, и в соответствии с этим порядком выставлять адреса при использовании linker counter `.`

Но как я уже сказал символ linker counter `.` выдаёт адреса для vma, а не lma! А ведь нам нужно знать также начало секции .data в FLASH, чтобы мы могли правильно в startup коде перенести данные из FLASH в RAM.

Для этого в linker script есть операция **LOADADDR(section_name)** которая возвращает адрес LMA для выбранной секции. Таким образом чтобы получить начальные адреса для LMA и VMA можно написать таким образом
```ld
SECTION 
{
    _sidata = LOADADDR(.data)
    .data :
    {
        _sdata = .;
        *(.data)
        _edata = .;
    }> RAM AT >FLASH
}
```
Таким образом мы получили начало .data во FLASH в символе `_sidata`, где s - start, i - initialization. А также мы получили начало .data в RAM в символе `_sdata` и конце в `_edata`. И таким образом в startup файле мы можем скопировать все данные из FLASH в RAM например при помощи такого кода:
```c
extern uint32_t _sidata;
extern uint32_t _sdata;
extern uint32_t _edata;

void init_data_section(void)
{
    uint32_t* flash_data = &_sidata;
    uint32_t* ram_data = &_sdata;
    
    while(dst < &_e_data)
    {
        *ram_data = *flash_data;
        flash_data += 1;
        ram_data += 1;
    }
}

```


## Продвинутые возможности

### ALIGN
Все примеры что были написаны выше на самом деле не будут работать корректно на всех системах, т.к. linker будет выравнивать все адреса по умолчанию, то есть по 64 бита или кратно 8. Однако на процессоре Cortex-M4 что стоит в stm32g4 используется 32 битная адресация(отсюда и название stm**32**), поэтому нам важно чтобы адреса были выровнены по 32 битам, или чтобы они были кратны 4. 

Для этого можно воспользоваться [Location counter](#Location%20counter%20and%20script%20symbols) и функцией ALIGN() которая будет выравнивать адреса по нужной кратности. Так например для `.data` секции правильным кодом в ld будет

```ld
.data :
{
    . = ALIGN(4);
    _sdata = .;
    *(.data)
    . = ALIGN(4);
    _edata = .;
} >RAM AT >FLASH
```

Первая ALIGN нужен чтобы начало `.data` было выровнено. А вторая ALIGN нужен чтобы конец секции тоже выровнять, чтобы следующая секция не начиналась «криво». Но лучше для каждой секции выставлять выравнивание, чтобы точно всё работало верно.

### KEEP
Линкер удаляет неиспользуемые функции и данные при ипользовании флага `--gc-section`, это полезная функция особенно на микроконтроллерах, где важна оптимизация памяти.

Однако такое может быть проблемой, потому что некоторые секции нужно оставить в любом случае, даже если кажется что оттуда что-то не используется. Так например с секцией для вектора прерываний. Он может не использоваться в коде, однако он нужен всегда, в любом случае он должен быть.

Для того чтобы линкер не удалял секции даже если кажется что они не используются, используется функция KEEP, так например для `.isr_vector`

```ld
SECTION 
{
    .isr_vector :
    {
        . = ALIGN(4);
        KEEP(*(.isr_vector)) /* Startup code */
        . = ALIGN(4);
    } >FLASH

}
```
Этот код сделает так, что `.isr_vector` секции будут в конечном коде всегда, не зависимо от того используются ли они в коде или нет

### NOLOAD
Это директива для секций, которая говорит линкеру что эта секция **не должна занимать место в образе прошивки**, т.е. линкер не будет записывать данные в итоговый ELF/HEX/SREC для этой секции. Однако линкер будет знать об этой секции, что она может использоваться во время выполнения в си коде, просто линкер скрипт в elf файле не будет туда ничего помещать, а только выделит для него адресное пространство.
Это полезно например для отладочной информации
```ld
SECTIONS
{
  .debug_info (NOLOAD) :
  {
    *(.debug_info)
  } >FLASH
}
```

### ALIAS
Эта директива создаёт псевдоним для уже существующего символа в линкер скрипте.
Это полезно для:
- **Обратной совместимости** со старым кодом, который использует старые имена символов.
- Создания **более описательных или понятных имён** для часто используемых адресов или областей памяти.

```ld
ALIAS(_legacy_heap_start, _heap_start);
```

### ASSERT
Эта директива работает примерно также как и assert в си, то есть она позволяет проверять условия, но в данном случае во время линковки
```ld
ASSERT(.text <= 64K, "Text section overflow - FLASH memory region is too small!");
```
Здесь  мы проверяем чтобы размер секции .text не превышал 64 КБ, иначе линкер выдаст ошибку "Text section overflow - FLASH memory region is too small!"


### /DISCARD/

Это специальный раздел памяти, который говорит линкеру не включать это в итоговый ELF. Так мы например можем убрать все секции из стандартных библиотек, такие как libc и libm. Ведь в них находятся функции которые требуют тех же системных вызовов, например `_read`, `_close`, `_exit`. Чтобы **не тянуть эти функции и не получать ошибки линковки**, с чем мы боролись в [[Как микроконтроллер запускает код]], линкер будет их просто отбрасывать.
Так можно использовать этот код:
```ld
/DISCARD/ :
{
    libc.a(*)
    libm.a(*)
    libgcc.a(*)
}
```
Обычно для микроконтроллеров используют флаг `--specs=nano.specs` которые подключают миниатюрную версию libc/libm где для каждой функции из стандартной библиотеки используется свой объектный файл(например для `sinf`, `cosf` и т.д.)
