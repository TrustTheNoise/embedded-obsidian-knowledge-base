# Для чего эта заметка

В этой заметке я попытаюсь описать полный цикл запуска микроконтроллера на примере stm32g4. То есть, опишу процесс компиляции, линковки, загрузки кода в микроконтроллер и собственно говоря запуска.

В отличие от ПК, где процесс запуска управляется BIOS и ОС, на микроконтроллере за инициализацию памяти, стека, копирование данных из Flash в SRAM и запуск `main()` отвечает стартовый код (startup), а за загрузку кода отвечает бутлоадер.

Полное понимание процесса запуска помогает лучше разбираться в работе RTOS и других библиотек, потому что они строятся на готовой инфраструктуре MCU: стек, глобальные данные, прерывания инициализируются стартовым кодом (`startup_stmxxxxxx.s`), на который RTOS опирается для своей работы.

# Создание исполняемого файла МК
Итак, мы разбирали процесс компиляции и линковки в современных ОС в [[Компилятор и линковщик]].  
Там мы узнали, что [компилятор](Компилятор%20и%20линковщик#Что%20делает%20компилятор?) переводит код на C в бинарный вид, понятный процессору, — то есть машинный код. То есть те же ассемблерные операции(ассемблер — это просто человеко-читаемые инструкции процессора).

Если у нас несколько файлов, которые должны взаимодействовать друг с другом в одном исполняемом файле, за «сборку» всего этого кода и данных, расстановку всех объектов в нужные секции памяти и формирование конечного `.exe` или `.elf` файла отвечает [линковщик](Компилятор%20и%20линковщик#Что%20делает%20Линковщик?).

То как выполняется код на компьтерах и/или более сложных устройств типа Raspberry PI, в общем на устройствах где есть ОС, можно прочитать тут [[Компилятор и линковщик#Роль ОС в запуске кода]], важно минимально понимать как это делает ОС, ведь в микроконтроллерах мы сами принимаем его роль чтобы сделать всё необходимое для запуска.

Итак, давайте тогда работать с кодом Си для микроконтроллеров. Понятное дело код никак не отличается, однако отличается компилятор, у нас другая платформа, а именно Cortex-M4 в stm32g4, так что нам нужно воспользоваться компилятором gcc для arm- [arm-none-eabi-gcc](https://developer.arm.com/downloads/-/gnu-rm)
Его название расшифровывается так:
- arm - ну для процессоров с архитектурой arm
- none - целевая операционная система, то есть мы говорим что нет ОС
- eabi - ABI(Application Binary Interface) для встраиваемых систем: Embedded ABI. Отличие от обычного ABI в том, что нет поддержки полного пользовательского окружения Linux, нет system calls, меньше накладных расходов, оптимизировано для встраиваемых процессоров.

Давайте напишем простейшую программу на Си:
```c title:main.c
int main(void) 
{
    return 0;
}
```

В отличии от обычного компиляции gcc для ОС, нам куда важнее выставить требуемый CPU, для которого будет компиляция. Т.к. для разных процессоров ARM для встраиваемых систем может сильно отличаться инструкции, поэтому мы выставим -mcpu=cortex-m4 для stm32g4, а также включим thumb инструкции, а также включим информацию для отладчиков через флаг -g

```bash
arm-none-eabi-gcc -mcpu=cortex-m4 -mthumb -g -O0 -c main.c -o main.o
```

Мы напоролись на гигантское количество ошибок линковки. Почему у нас просят кучу различных системных функций типа `_close`, `_write` и т.д. рассказывалось в [[Компилятор и линковщик#Роль ОС в процессе линковки]], ведь функции `_close_r`, `_write_r` и т.д. из-за которых у нас и просят системные вызовы, это функции стандартной библиотеки С, которые уже используются в других функциях библиотеки.

Нам нужно все эти системные вызовы самим создать, например при создании cubeide проекта будет автоматически создаваться файл со всеми syscalls заглушками. Мы же создадим заглушки через `--specs=nosys.specs` которая сделает для всех системных вызовов заглушки которые будут просто вызывать ошибки. Однако это не решение, ведь правильнее будет сделать системные вызовы самому ручками, чтобы они правильно всё выполняли.

Из-за этого у нас будут предупреждения а не ошибки и это нам позволит создать .elf файл
