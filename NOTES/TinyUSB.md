# TinyUSB

TinyUSB — это открытая, кроссплатформенная библиотека USB. Она не привязана к конкретной MCU.
А благодаря уровню абстракции аппаратной платформы (board/MCU abstraction layer) , можно писать код TinyUSB который может работать на разных микроконтроллерах без изменений(кроме настройки самого abstraction layer)
Библиотека поддерживает работу как **USB Device**, так и **USB Host**, а также USB OTG, что позволяет MCU переключаться между режимами хоста и устройства.

Главная особенность TinyUSB — безопасность для памяти и потокобезопасность без использования динамического выделения. Библиотека спроектирована таким образом, что все события, возникающие в прерываниях USB (например, приход пакета данных, подключение или отключение шины), **не обрабатываются напрямую в ISR**. Вместо этого в обработчике прерываний просто **устанавливаются флаги событий или заполняются буферы**, а реальная обработка этих событий выполняется в функции задачи TinyUSB (`tud_task()`), которая вызывается в основном цикле программы (`while(1)` в `main`) либо из задачи RTOS, если она используется.

Для сравнения как это всё работает в [[USB HAL]] от STM и как это реализовано в TinyUSB 

В отличие от USB HAL, где передача и приём данных выполняются напрямую через аппаратные FIFO (Tx/Rx FIFO) контроллера OTG_FS, в TinyUSB используется **программный буфер** как промежуточное звено между приложением и USB-периферией.

# Настройка TinyUSB

> [!Sources] 
>  [Port config/init confusion, whats the recomended way? · hathach/tinyusb · Discussion #2558 · GitHub](https://github.com/hathach/tinyusb/discussions/2558)

В последних версиях **TinyUSB** изменилась система конфигурации. Ранее требовалось задавать макросы вроде `CFG_TUSB_RHPORT0_MODE` или `CFG_TUSB_RHPORT1_MODE`, чтобы указать, какой **Root Hub Port (RHPORT)** использовать и в каком режиме (Device / Host, Full-/High-Speed)

Это делалось например таким образом
```c
// выставляем для Root hub port 0 режим Full-speed устройста 
#define CFG_TUSB_RHPORT0_MODE   (OPT_MODE_DEVICE | OPT_MODE_FULL_SPEED)
```

Далее конфигурация всего TinyUSB библиотеки происходила в `tusb_init` которая смотрела на дефайны `CFG_TUSB_RHPORT0_MODE/CFG_TUSB_RHPORT1_MODE` и на их основе настраивала библиотеку

> [!help] 
>  **Root Hub Port (RHPORT)** — физический USB-порт микроконтроллера. На STM32 FS-порт OTG_FS обычно соответствует RHPORT0, HS-порт OTG_HS — RHPORT1.

Теперь в TinyUSB вместо этих устаревших макросов каждый порт настраивается индивидуально. Так появились две функции инициализации: `tud_init(uint8_t rhport)` для настройки rhport в режим устройства и `tuh_init(uint8_t rhport)` для настройки rhport в режим хоста.

Такой подход позволяет динамически менять режим работы порта, например через `tusb_deinit(rhport)` перед новой инициализацией.

В новых примерах TinyUSB для удобства появились макросы:
- `BOARD_TUD_RHPORT` / `BOARD_TUH_RHPORT` — указывают, какой RHPORT используется на плате для Device / Host. Используются программистом при вызове `tud_init` / `tuh_init`.
- `BOARD_TUD_MAX_SPEED` / `BOARD_TUH_MAX_SPEED` — задают желаемую скорость работы порта, которая через `CFG_TUD_MAX_SPEED` / `CFG_TUH_MAX_SPEED` используется самой библиотекой при инициализации в `tud_init/tuh_init`

Таким образом, макросы `BOARD_*` упрощают перенос проекта между платами, а `CFG_*` управляют внутренней настройкой TinyUSB.

## Структура данных для каждого интерфейса CDC 

В TinyUSB для каждого CDC-интерфейса создаётся своя структура `cdcd_interface_t`, и они хранятся в глобальном массиве `_cdcd_itf[CFG_TUD_CDC]`. Каждая структура содержит **свой** TX-поток (`tx_stream`) и **свой** RX-поток (`rx_stream`), а также отдельные буферы FIFO.

```c
typedef struct {
  uint8_t rhport;               // Номер Root Hub Port
  uint8_t itf_num;              // Порядковый номер интерфейса в конфигурации USB
  uint8_t ep_notify;            // Номер эндпоинта уведомлений (Interrupt In) для передачи состояния
  uint8_t line_state;           // Состояние сигналов управления (Bit 0: DTR, Bit 1: RTS)

  /*------------- С этого момента данные не очищаются при сбросе шины (Bus Reset) -------------*/
  TU_ATTR_ALIGNED(4) cdc_line_coding_t line_coding; ///< Настройки порта: скорость, четность, стоп-биты
  char wanted_char;             // Символ-маркер, при получении которого вызывается коллбэк

  tu_edpt_stream_t tx_stream;   // Абстракция потока для передачи данных (TX Stream)
  tu_edpt_stream_t rx_stream;   // Абстракция потока для приема данных (RX Stream)

  uint8_t tx_ff_buf[CFG_TUD_CDC_TX_BUFSIZE]; // Внутренний кольцевой FIFO буфер для передачи
  uint8_t rx_ff_buf[CFG_TUD_CDC_RX_BUFSIZE]; // Внутренний кольцевой FIFO буфер для приема
} cdcd_interface_t;
```
Таким образом, интерфейс `itf=0` и `itf=1` полностью изолированы друг от друга — у каждого своя TX FIFO (`tx_ff_buf`), свой `ep_buf`, своя конечная точка USB (endpoint).

## Объяснение передачи данных в CDC

> [!Source] 
> [Диалог с нейронкой о том как работает CDC TX](https://deepwiki.com/search/-cdc-tudcdcnwrite-tudcdcnwrite_23c016a5-4e3d-4862-88e7-c9edd20a87a9) 

- [ ] #todo Переделать эту часть с учётом того что Bulk transfer может занимать до 19 мест в USB кадре, поэтому можно изменять CFG_TUD_CDC_EP_BUFSIZE, значит информация о том что мы можем за раз передавать только 64 байта неверная

Итак у каждого интерфейса есть свой внутренний кольцевой FIFO буффер для передачи `tx_ff_buf` размер которого определяет дефайн `CFG_TUD_CDC_TX_BUFSIZE`, управление этим буффером производит `tx_stream` со структурой
```c
typedef struct {
uint8_t  hwid;    // Идентификатор контроллера (rhport для устройства или daddr для хоста)
bool     is_host; // Флаг режима работы: 1 — USB Host, 0 — USB Device
uint8_t  ep_addr; // Физический адрес USB-эндпоинта (включая направление: IN/OUT)
uint16_t mps;     // Максимальный размер пакета (Max Packet Size) для данной конечной точки

uint16_t ep_bufsize;// Размер линейного буфера (ep_buf). Размер 
                    // задается через CFG_TUD_CDC_EP_BUFSIZE. 
uint8_t  *ep_buf;   // Линейный буфер (адрес в RAM). Используется 
                    // для прямого обмена с аппаратным буффером
                    // в USB контроллере на МК.
tu_fifo_t ff;       // Кольцевой FIFO, использующая tx_ff_buf или
                    // rx_ff_buf, вторая буфферизация к которому есть
                    //доступ у пользователя, туда идут данные из ep_buf 
                    // настраивается через CFG_TUD_CDC_(T/R)X_BUFSIZE
} tu_edpt_stream_t;
```

При вызове `tud_cdc_n_write(itf, buffer, bufsize)` мы записываем данные в кольцевой буффер `tx_stream->ff`(в котором находится ссылка на tx_ff_buf), и если мы записали в FIFO >= MPS байт (MPS = Max Packet Size, обычно 64 байта для Full Speed или 512 для High Speed) тогда мы сразу вызываем передачу этих MPS байт из FIFO на endpoint buffer(то есть на аппаратный EP-буффер), и **сразу говорим usb периферий, что если хост попросит данные то usb периферия отправит эти данные из endpoint buffer**

> [!important] 
>  То есть, если ты пишешь меньше MPS байт в tud_cdc_n_write - тогда данные просто сохраняться в FIFO, но отправка не начнётся. Соответственно чтобы гарантированно отправить данные (даже если их меньше MPS), нужно вызвать `tud_cdc_n_write_flush(itf)`

Когда USB-контроллер завершил передачу данных хосту, **конкретно передачу аппаратного `ep_buf`, а не всех данных в `tx_stream->ff`**, тогда вызывается прерывание, где событие конца передачи становится в очередь событий. И когда в `tud_task()` доходит очередь до события конца передачи выщывается  callback `cdcd_xfer_cb`. Которая в свою очередь:
1. Вызывает callback пользователя `tud_cdc_tx_complete_cb(itf)`. Это позволяет на лету подбросить в FIFO данные, которые будут подхватываться дальше в пункте два
2. Если есть данные в `tx_stream->ff` тогда автоматически пытается отправить оставшиеся данные из FIFO.
3. Если данных больше нет и при этом предыдущий пакет был равен MPS(максимальному размеру пакета) тогда в соответствии с USB 2.0. specification он отправляет **ZLP (Zero Length Packet)** через `tu_edpt_stream_write_zlp_if_needed` чтобы показать что отправка закончилась.

## Объяснение приёма данных в CDC

> [!Source] 
> [Диалог с нейронкой о том как работает CDC RX](https://deepwiki.com/search/-cdc-usb-tinyusb-callback-tudc_41f201f3-8e56-473e-a988-fd1192931ea1) 
> [Диалог с нейронкой про то что Bulk может занимать до 19 мест в кадре](https://deepwiki.com/search/-cfgtudcdcepbufsize-64-bulk-tr_d9355a2b-1ad5-47e0-b996-bff789c83341)

> [!caution] 
> Эта часть написана конкретно с оглядкой на то как работает драйвер в TinyUSB для stm32_fsdev контроллер
> По идее разница с dwc2 контроллером(что например стоит в stm32f4) минимальны и заключаются только в том, что в dwc2 вместо PMA используется аппаратный FIFO, то есть в случае dwc2 нам не нужно перенастраивать PMA для приёма новых данных, в остальном разницы нет. 

В стеке **TinyUSB** для контроллеров **STM32 (FSDEV)** используется многоуровневая система буферизации для обеспечения максимальной пропускной способности и асинхронности:

1. **PMA (Packet Memory Area):** Специализированная аппаратная область памяти USB-периферии объемом 1 КБ. Это «первая миля», куда физический уровень (PHY) укладывает пакеты по 64 байта.
2. **Линейный софтверный буфер (`ep_buf`):** Промежуточный транзитный буфер в SRAM. Он необходим для **сборки (batching)** нескольких USB-пакетов в одну логическую транзакцию. Это позволяет драйверу накопить, например, 256 байт (4 пакета по 64) и передать их выше по стеку одним событием, минимизируя нагрузку на системную очередь событий и исключая частые переключения контекста.
3. **Кольцевой FIFO-буфер(`rx_ff_buf` в `cdcd_interface_t` или `ff` в `tu_edpt_stream_t`)**: Финальный программный буфер в SRAM. Туда данные переходят из ep_buf, и к FIFO-буферу приложение имеет прямой доступ через API (`tud_cdc_read`).

```c
typedef struct {
uint8_t  hwid;    // Идентификатор контроллера (rhport для устройства или daddr для хоста)
bool     is_host; // Флаг режима работы: 1 — USB Host, 0 — USB Device
uint8_t  ep_addr; // Физический адрес USB-эндпоинта (включая направление: IN/OUT)
uint16_t mps;     // Максимальный размер пакета (Max Packet Size) для данной конечной точки

uint16_t ep_bufsize;// Размер линейного буфера (ep_buf). Размер 
                    // задается через CFG_TUD_CDC_EP_BUFSIZE. 
uint8_t  *ep_buf;   // Линейный буфер. Используется 
                    // для сборки нескольких USB-пакетов в одну
                    // логическую транзакцию
tu_fifo_t ff;       // Кольцевой FIFO, использующая tx_ff_buf или
                    // rx_ff_buf, вторая буфферизация к которому есть
                    //доступ у пользователя, туда идут данные из ep_buf 
                    // настраивается через CFG_TUD_CDC_(T/R)X_BUFSIZE
} tu_edpt_stream_t;
```

Размер выделяемый в PMA-буфере = `packet_size` из дескриптора для endpoint'а, например для Full Speed Bulk Transfer может быть с `packet_size` 8, 16, 32, 64 байта. Выделение происходит единожды при открытии эндпоинта.

Итак давайте разберёмся как происходит приём данных через CDC. 
Сразу при открытии транзакции мы разрешаем приём данных через endpoint, делается это через `tu_edpt_stream_read_xfer` в котором вычисляется сколько данных мы готовы принять "сразу" у хоста. Это значение вычисляется как `total_len` для трансфера.
Оно высчитывается таким образом
```c
// Проверяем остаток места в кольцевом буфере (rx_ff_buf)
uint16_t available = tu_fifo_remaining(&s->ff);

// Принимаем данные если влезет хотя бы один полный пакет (64 байта)
if (available >= s->mps) 
{
    // Округляем место в FIFO вниз до кратного 64 (целое число пакетов)
    uint16_t count = (uint16_t) (available & ~(s->mps - 1));
    // Ограничиваем запрос размером линейного буфера ep_buf
    count = tu_min16(count, s->ep_bufsize);
    // Разрешаем приём данных через endpoint
    TU_ASSERT(stream_xfer(s, count), 0);
    return count;
} else 
{
    // Места нет — переводим эндпоинт в режим NAK (пауза для хоста)  
    stream_release(s);
    return 0;
}
```

> [!summary] 
> То есть если мы увеличим CFG_TUD_CDC_EP_BUFSIZE тогда увеличиться и `s->ep_bufsize` и таким образом будет возможность принимать куда больше данных разом

Когда происходит Bulk transaction от хоста(который может быть с максимум 64 байтами данных в Full speed), то происходит прерывание в обработчике которого копируются данные из PMA буффера в линейный софтверный буфер `ep_buf`.  В этом же обработчике прерываний есть такой код:
```c
if ((rx_count < xfer->max_packet_size) // пришёл пакет меньше 64 байт
    || (xfer->queued_len >= xfer->total_len)) // пришло ep_bufsize байт
{
    // Фиксируем конец логической транзакции

    // Сбрасываем лимит приема в PMA до стандартных 64 байт 
    // (защита от переполнения)
    btable_set_rx_bufsize(ep_id, BTABLE_BUF_RX, xfer->max_packet_size);
     
    // Добавляем в event queue событие конца приёма
    dcd_event_xfer_complete(0, ep_num, xfer->queued_len, XFER_RESULT_SUCCESS, true);
 
    // Сбрасываем счетчики для следующего трансфера
    xfer->total_len = xfer->queued_len = 0;
  } else {
    // ТРАНЗАКЦИЯ НЕ ЗАВЕРШЕНА: разрешаем endpoint принять новый пакет
    if (!is_iso) {
        // Рассчитываем размер следующей порции 
        // (остаток или полный пакет 64 байта)
        const uint16_t cnt = tu_min16(xfer->total_len 
                - xfer->queued_len, xfer->max_packet_size);
        // Настраиваем PMA буффер на прием этой порции
        btable_set_rx_bufsize(ep_id, BTABLE_BUF_RX, cnt);
    }
    // Переключаем статус эндпоинта обратно в VALID (активен)
    ep_reg &= USB_EPREG_MASK | EP_STAT_MASK(TUSB_DIR_OUT); 
    ep_change_status(&ep_reg, TUSB_DIR_OUT, EP_STAT_VALID);
    
    // Записываем команду в регистр: 
    // теперь МК примет следующий пакет автоматически
    ep_write(ep_id, ep_reg, false);
}
```

То есть только если мы примем столько пакетов сколько мы рассчитали в total_len(либо мы приняли неполный пакет, что в USB 2.0. говорит что логическая транзакция завершена), тогда формируется событие `DCD_EVENT_XFER_COMPLETE` что помещается в очередь событий `queue_event`. Поэтому обработка события является отложенной, реальная обработка события что пришли данные происходят в `tud_task()`.

Когда `tud_task()` обрабатывает событие полученния данных для RX-эндпоинта, вызывается `cdcd_xfer_cb` который:
1. Копирует данные из `ep_buf` в кольцевой FIFO `rx_ff_buf`
2. Если пользователь установил символ через `tud_cdc_n_set_wanted_char()`, драйвер ищет его в только что принятых данных и при нахождении вызывает `tud_cdc_rx_wanted_cb`
3. Если в FIFO есть данные (т.е. только что принятый пакет не пустой), вызывается пользовательский callback `tud_cdc_rx_cb(itf)`.
4. После этого вызывается `tu_edpt_stream_read_xfer(stream_rx)` — в нём проверяется есть ли в `rx_ff_buf` место >= MPS(Maximum Packet Size), и если да, то вызывается `usbd_edpt_xfer`, который в свою очередь вызывает `dcd_edpt_xfer` в котором рассчитывается `total_len` для нового ожидаемого трансфера и эндпоинт снова «заряжается», чтобы быть готовым принять следующий трансфер от хоста.

> [!important] 
> Однако в `tud_cdc_rx_cb(itf)` обычно приложение вызывает `tud_cdc_n_read(itf, buffer, bufsize)` которая читает байты из FIFO `rx_stream.ff` и возвращает <= bufsize данных в зависимости от того есть ли в FIFO столько данных. И при этом она тоже вызывает `tu_edpt_stream_read_xfer` - ещё одна попытка перезарядить эндпоинт, если в FIFO освободилось место. Поэтому ещё во время пользовательского callback `tud_cdc_rx_cb(itf)` может перезарядиться эндпоинт и в ep_buf могут сразу быть записаны новые данные.

# TinyUSB Audio

## Переключение alternate settings
Изначально все endpoint'ы для аудио должны быть отключены в alternate settings 0 для интерфейса. Только когда хост активирует другой alternate settings в интерфейсе где уже есть endpoint'ы, только тогда USB-контроллер сможет принимать данные. Таким образом сначала хост отправляет `SET_INTERFACE` с `bAlternateSetting = 1`, когда мы примем SETUP пакет вызовется прерывание в котором событие `DCD_EVENT_SETUP_RECEIVED` поместится в очередь событий которая будет обработана только в tud_task() который вызовет `audiod_set_interface()` где:

1. Закрывается предыдущий EP
2. Открываются новые EP которые используются в данном alternate settings.
3. Если в Alternate settings есть EP OUT тогда сразу же отправляется первая "заявка" на приём у USB-контроллера на МК через `usbd_edpt_xfer()`

После этого вызова USB-контроллер знает: "следующий пакет от хоста — помести в `lin_buf_out`".

## Объяснение приёма аудио данных

> [!source] 
>  [Диалог с нейронкой по работе приёма в TinyUSB Audio](https://deepwiki.com/search/-audio-interface-pcm-16-tudtas_000611a6-6e5c-412b-9249-c718d8a4be15?mode=deep)

У audio функции для endpoint OUT есть кольцевой software FIFO `ep_out_ff` размер которого определяется через `CFG_TUD_AUDIO_FUNC_X_EP_OUT_SW_BUF_SZ`, где X - номер аудио интерфейса в устройстве, а также есть `lin_buf_out` линейный аппаратный буффер для Endpoint OUT в USB контроллере, его размер определеятся через `CFG_TUD_AUDIO_FUNC_X_EP_OUT_SZ_MAX`.

Хост отправляет данные, например если у нас 48000 Гц частота дискретизации и при этом используется PCM 16 бит на 2 канала, тогда хост будет отправлять каждый кадр(1 мс для Full Speed) по 48 сэмплов × 2 канала × 2 байта = 192 байт/фрейм.
Когда USB-контроллер принимает эти данные, то он вызывает прерывание, где для события `DCD_EVENT_XFER_COMPLETE` проверяется есть ли у драйвера `xfer_isr`то есть должен ли он что-то делать с этим событием в прерывании. 

> [!important] 
> И тут важный момент, рассмотрим два драйвера TinyUSB для понимания различия между ними:
> - в [[#TinyUSB CDC]] используется Асинхронная модель. У CDC-драйвера `xfer_isr` равен `NULL`. Поэтому при событии `DCD_EVENT_XFER_COMPLETE` в прерывании мы просто помещаем это событие в очередь событий, чтобы обработка принятых данных происходила в `tud_tas()`. Это делает обработку «мягкой» и безопасной для длинных операций в приложении.
> - В TinyUSB Audio используется потоковая модель, поэтому у Audio определён `audiod_xfer_isr`. Это позволяет обрабатывать данные непосредственно внутри прерывания. `tud_task()` вообще никак не участвует. 

`audio_xfer_isr()` находит, к какому audio-интерфейсу принадлежит endpoint, и для OUT пакета вызывает `audiod_rx_xfer_isr()`. 

`audiod_rx_xfer_isr()` делает основную работу
1. Копирует полученные данные из линейного буфера `lin_buf_out` в кольцевой software FIFO `ep_out_ff`.
2. Сразу же выставляется новая заявка на следующий пакет — контроллер снова будет ждать следующего пакета от хоста.
3. Если используется AUDIO_FEEDBACK_METHOD_FIFO_COUNT, в таком случае мы обновляем счётчик данных для feedback. Про это рассказано в [[#Feedback control]]
4. Вызывается пользовательский callback `tud_audio_rx_done_isr()`. Этот callback — возможность среагировать немедленно, например, выставить флаг для I2S DMA, или прочитать данные прямо из ISR.


## Feedback control
При разборе [синхронизации USB Audio](Синхронизация%20USB%20Audio.md) что тактирование USB-устройства, принимающего аудиоданные от хоста, изначально не синхронизировано с тактированием хоста (так называемым [Service Clock](Синхронизация%20USB%20Audio.md#Синхронизация%20USB%20Audio)). Это связано с тем, что устройство обычно использует собственный генератор, частота которого неизбежно отличается от частоты хоста из-за допусков генераторов и температурного дрейфа.

Если устройство использует внутренний аудиотакт, который не может подстраиваться под хост (асинхронный режим работы), возникает необходимость синхронизировать поток данных со стороны хоста. В противном случае разница частот приведёт к постепенному переполнению или опустошению аудиобуфера устройства.

Для решения этой задачи используется [isochronous feedback endpoint](Синхронизация%20USB%20Audio#Isochronous%20feedback%20endpoint). Через него устройство периодически сообщает хосту фактическую скорость потребления аудиоданных, то есть количество сэмплов, которое устройство должно получать за каждый изохронный USB-кадр. Хост, получая эту информацию, корректирует скорость передачи данных, чтобы поддерживать непрерывность аудиопотока.

В TinyUSB реализовано три метода вычисления значения обратной связи (feedback), которые определяют, какое количество сэмплов устройство будет запрашивать у хоста для поддержания стабильного аудиопотока.

### AUDIO_FEEDBACK_METHOD_FIFO_COUNT
Метод `AUDIO_FEEDBACK_METHOD_FIFO_COUNT` вычисляет значение isochronous feedback на основе степени заполнения приёмного FIFO-буфера аудиоданных.

Идея метода заключается в том, что FIFO используется как индикатор рассогласования частот между хостом и устройством. Если хост передаёт данные быстрее, чем устройство их воспроизводит, уровень заполнения FIFO будет расти. Если медленнее — FIFO будет постепенно опустошаться. TinyUSB отслеживает уровень заполнения FIFO и корректирует значение feedback таким образом, чтобы удерживать буфер примерно наполовину заполненным.

Вычисление нужного audio feedback остаётся за самим стеком TinyUSB, всё что нужно сделать программисту это в callback `tud_audio_feedback_params_cb()` указать:

```c
void tud_audio_feedback_params_cb(uint8_t func_id, 
uint8_t alt_itf, audio_feedback_params_t *feedback_param)
{
    feedback_param->method = AUDIO_FEEDBACK_METHOD_FIFO_COUNT;
    feedback_param->sample_freq = current_sample_rate; 
}
```

После этого callback'а TinyUSB:
1. Отслеживает количество байт во внутреннем FIFO для Iso OUT endpoint'a размер которого задаётся через `CFG_TUD_AUDIO_FUNC_1_EP_OUT_SW_BUF_SZ` в `tusb_config.h` и высчитывает 
2. Вычисляет отклонение текущего уровня FIFO от заданного порога. Причём заданный порог можно выставить через `feedback_param->fifo_count.fifo_threshold` если не выставить его вручную в `tud_audio_feedback_params_cb()` то он будет выставлен на значение по умолчанию `CFG_TUD_AUDIO_FUNC_1_EP_OUT_SW_BUF_SZ/2`
3. На основе вычисленного отклонения FIFO TinyUSB формирует значение feedback endpoint.
4. Передаёт feedback хосту, который корректирует количество отправляемых сэмплов.

> [!warning] 
> Недостатки метода:
> -  Требуется относительно большой FIFO. TinyUSB рекомендует использовать размер FIFO не менее четырёх USB-фреймов.
> - Добавляется задержка аудиопотока в половину глубины FIFO
> - Метод реагирует на рассогласование частот медленнее, чем методы, основанные на измерении мастер-такта.

### FREQUENCY_FIXED / FREQUENCY_FLOAT

В этом методе TinyUSB пытается напрямую измерять реальную скорость, с которой устройство воспроизводит или генерирует аудиосэмплы, и уже из этой скорости вычислять значение feedback, которое отправляется USB-хосту. То есть это метод о котором рассказывается в USB 2.0. specification Главе 5.12. 

В основе лежит предположение, что внутри устройства есть стабильный мастер-клок $f_(m)$. Например PLL с HSE в stm32. Из этого мастер-клока через делители мы получаем частоту дискретизации $f_(s)$.

И в этом методе TinyUSB не измеряет сами аудиосэмплы — он измеряет, сколько тактов мастер-клока прошло за один USB-кадр. Поскольку USB аудио синхронизируется по кадрам SOF, стек использует прерывание SOF как точку измерения.
