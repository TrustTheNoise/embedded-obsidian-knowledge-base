# CANopen и с чем его едят
CANopen это основанный на [[CAN шина]] протокол верхнего уровня модели [[OSI]]. Его полезность в том что он обеспечивает совместимость "из коробки" между устройствами (узлами) в промышленном оборудовании. Кроме того, он предоставляет стандартные методы для настройки устройств, в том числе после установки.

## CANopen и OSI

Ведь просто протокол CAN обеспечивает только [Физический](Физический%20уровень%20OSI) и [Канальный](Канальный%20уровень%20OSI) уровни. То есть CAN определяет то какие будут электрические сигналы и устройства обеспечивающие эти сигналы, а также определяет как будут передаваться собственно данные и как будет производится контроль ошибок. Он отлично подходит для быстрой и надежной передачи данных между устройствами, но **не описывает**, что означают передаваемые данные, как их интерпретировать или как настраивать устройства.

Собственно говоря смысл же сети в том, чтобы мы могли управлять несколькими устройствами и при этом у нас был стандартизированный вариант данных, чтобы каждый из этих устройств понимали друг-друга. По сему нам и понадобился CANopen - это протокол верхнего уровня, который добавляет:
- Стандартизованный формат данных
- Единые правила конфигурации и инициализации устройств
- Механизмы обмена служебной информацией 
- Сетевые службы

В оригинальном CAN нету ни мастеров ни слейвов. Все равны на шине. Однако в CANopen всё же есть условный мастер, который может допустим управлять состоянием других узлов.

Давайте-же разберёмся что конкретно надобавлял CANopen с точки зрения [[OSI]] уже с терминологией из CANopen, которая будет объясняется в [[CANopen Application layer]]
![[Pasted image 20250609122731.png|350]]

- Физический уровень определяется [[CAN шина#Физический уровень]] т.е. используются доминантные/рецессивные биты. В случае использования электрического сигнала мы используем трансиверы с дифференциальным сигналом.

- Канальный уровень тоже определяется [[CAN шина#Канальный уровень]] то есть обнаружение ошибок через CRC, bit stuffing и прочим, подтверждается передача

Остальные уровни точечно(но не полностью в соответствии с ISO/OSI) выполняет именно CANopen

- Сетевой уровень даёт стандартизированные COB-ID для различных функций. Даёт конфигурировать узел мастером через SDO

- Транспортный уровень частично обеспечивается через NMT(что такое этот транспортный уровень??)

- Сеансовый уровень выполняется через SDO, т.к. узел должен отправить мастеру запрос на чтение SDO из общей базы данных(объектного словаря мастера)

- Уровень представления - сама база данных сети представляется через объектный словарь с определёнными CANopen типами данных.

- Прикладной уровень - ну это уже мы ручками пишем используя фреймворки типа [[CANopenNode]]

# Разбор CANopen
У CiA просто гигантское количество документаций описывающие разные слои CANopen, а также дополнительные функции, которые могут использоваться, а могут не использоваться. Для каждого документа ~~возможно~~ будет создана своя заметка

- [CiA 301 application layer](CANopen%20Application%20layer) описывает
	- структуру и назначение Object Dictionary (словаря объектов);
	- работу NMT (Network Management) — управление состояниями узлов (старт, стоп, предоперационный режим и т.д.);
	- форматы и обработку сервисных сообщений (SDO, PDO, Heartbeat, Node Guarding);
	- правила передачи данных и обмена между узлами в сети;
	- типы сообщений, их структуру и порядок взаимодействия;
	- основные требования к устройствам, чтобы они могли корректно работать в CANopen-сети.
