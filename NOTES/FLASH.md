# Задержка доступа для чтения
Чтобы правильно считывать данные из Flash-памяти, количество циклов ожидания на чтение данных (LATENCY) должно быть правильно запрограммировано в регистре управления доступом к Flash-памяти (FLASH_ACR) в соответствии с частотой тактового сигнала процессора (HCLK) и диапазоном внутреннего напряжения устройства VCORE.
![[Pasted image 20250408130940.png]]
> [!hint] 
>  После сброса частота тактового сигнала процессора составляет 16 МГц с режимом $V_{CORE}$ Range 1 normal mode. Соответственно после сброса LATENCY установлено в 0 WS(1 такт ожидания)

Вообще по идее LATENCY влияет на то сколько циклов будет занимать операция ldr в ассемблере, но только когда мы обращаемся к адресам FLASH памяти, ведь ldr можно использовать для RAM памяти.

Для изменения LATENCY выполните следующие шаги:
1) Запрограммируйте новое количество циклов ожидания в биты LATENCY регистра управления доступом к Flash-памяти (FLASH_ACR).
2) Убедитесь, что новое количество циклов ожидания учитывается при доступе к Flash-памяти, прочитав регистр FLASH_ACR.
```c
// Установить задержку на доступ к Flash 
FLASH->ACR &= ~FLASH_ACR_LATENCY_Msk; 
FLASH->ACR |= FLASH_ACR_LATENCY_2WS;
while((FLASH->ACR & FLASH_ACR_LATENCY_Msk) != FLASH_ACR_LATENCY_2WS) {}
```
> [!question] 
> Но вообще я не уверен что нужно прям обязательно проверять что новое количество циклов ожидания выставилось. Во всяком случае в наших проектах мы это не делали и ничего не сломалось. Да и будем честны оно 300 раз установится пока мы ещё будем изменять частоту тактирования

# Структура FLASH 
FLASH состоит из банки(или банок) памяти для различных данных(но в основном там хранится сама программа которая запускается на микроконтроллере, но можно засунуть и свои данные), а также из информационного блока который состоит из:
- System memory из которой запускается сам микроконтроллер. В этой памяти находится бутлоадер. 
- 1 кбайтный OTP(one-time programmable) которая не может быть очищено и может быть перезаписана 1 раз. Даже если мы запишем только один бит, даже если это будет 0, то всё двойное слово(64 бита, т.к. слово в stm32 это 32 бита) больше нельзя будет переписать.
- Option bytes которая нужна для конфигурации. В неё записывается уровень защиты памяти и подобное.

![[Pasted image 20250410191622.png|550]]

![[Pasted image 20250410191631.png|550]]

Как видно из данных таблиц можно работать с памятью через один банк или два банка. В случае одного банка у нас будет страница по 4 кбайта, а в случае двух банок у нас будет страница по 2 кбайта.
Однако такие таблицы(и работа с двумя банками памяти) доступна только для category 3 devices(таблицу категорий можно посмотреть в самом начале заметки)

Разница структуры в других категориях в том, что в category 2 и 4 нет управления памятью через 2 банки. Поэтому там 1 банка причём с размером страниц в 2 кбайта. А также они слегка отличаются размерами information block.

# Исправление кода ошибки (ECC)
В работе с FLASH есть процесс нахождения(и исправления) ошибок при чтении/записи. Соответственно если происходит ошибка при чтении или записи то это может заметить микроконтроллер и даже, если у нас ошибка всего в 1 бите, то он может её сходу исправить. Если же ошибки в двух битах, то микроконтроллер не сможет их сходу исправить, а только уведомит об ошибках.
## DBANK=1, 64 бита длина данных
В таком случае данные в Flash с размером 72 бита (64 битов самих данных и 8 битов на каждые данные). 

Соответственно если при записи/чтении произошла ошибка в 1 бите, то микроконтроллер автоматически её исправляет и отправляет/принимает правильные данные, при этом он поднимает флаг ECCC в FLASH_ECCR. Если поднят флаг ECCCIE то в таком случае будет сгенерировано прерывание.

Если же зарегистрированы ошибки в двух битах то поднимается флаг ECCD в FLASH_ECCR. И, главное, генерируется NMI прерывание.
> [!help] 
>  NMI это прерывание которое нельзя деактивировать. Оно будет происходить в любом случае. То есть всегда когда генерируется NMI прерывание мы переходим в обработчик прерывания NMI. Он нужен для критических ошибок(типа ошибок в 2х битах при чтении/записи FLASH)

При этом если возникнет ошибка ECCC или ECCD адрес двойного слова в которой случилась ошибка находится в ADDR_ECC[20:0], а номер банки в которой случилась ошибка в BK_ECC в регистре FLASH_ECCR. 
> [!info] 
> ADDR_ECC[2:0] всегда очищена. Т.к. у нас побайтная адресация, а $2^3=8$ то есть у нас в ADDR_ECC будет адрес на первый байт в двойном слове, а это как раз и есть 8 адресов, т.к. $8 \text{ адресов}\cdot 8 \text{ битов в байте} = 64 \text{ битов}$

> [!warning] 
>  Если ECCC или ECCD установлены, то ADDR_ECC и BK_ECC не будут обновляться даже если появится новая ошибка при чтении/записи. Регистр FLASH_ECCR обновляется только если ECCC и ECCD очищены

## DBANK=0, 128 битов длина данных
В таком случае естественно мы пользуемся памятью как 1 банкой, где у нас 4 кбайта на страницу и 144 битные данные( два 64 битных слова и на каждое двойное слово 8 битов, итого 64+8+64+8=144).
Механизм ECC поддерживает:
- Обнаружение и исправление ошибки в одном бите
- Обнаружение ошибки в двух битах на каждые 64 битные слова.

Для начала нужно проверить SYSF_ECC бит, который отвечает за ошибки в System memory и если он выставлен, нужно работать с данными также как если бы у нас было бы [DBANK=1](#DBANK=1,%2064%20бита%20длина%20данных). Так как конкретно область System memory всегда будет из двух банков. Если же ошибка произошла в main memory то всё будет работать по следующему сценарию

Когда обнаружена одна ошибка в 64 младших битах (биты 63:0) и исправлена, устанавливается флаг ECCC (ECC correction) в регистре FLASH_ECCR. Если же ошибка найдена и исправлена в 64 старших битах (биты 127:64), устанавливается флаг ECCC2 (ECC2 correction) в регистре FLASH_ECCR.

Если установлен ECCCIE, генерируется прерывание. Пользователь должен прочитать ECCC и ECCC2, чтобы узнать, какая часть 128-битных данных была исправлена (либо 63:0, либо 127:64, либо обе).

Когда обнаружены две ошибки в 64 младших битах, устанавливается флаг ECCD (ECC detection) в регистре FLASH_ECCR. Если же две ошибки в 64 старших битах (биты 127:64), устанавливается флаг ECCD2 (ECC2 detection) в регистре FLASH_ECCR. В этом случае генерируется NMI (Non-Maskable Interrupt). Пользователь должен прочитать ECCD и ECCD2, чтобы узнать, в какой части 128-битных данных обнаружена ошибка (либо 63:0, либо 127:64, либо обе).

Когда обнаружена ошибка ECC, адрес неисправного двухкратного двойного слова сохраняется в ADDR_ECC[20:0] в регистре FLASH_ECCR. ADDR_ECC[20:0] содержит адрес двухкратного двойного слова.
> [!info] 
>  ADDR_ECC[3:0] всегда очищена. Т.к. у нас побайтная адресация, а $2^4=16$ то есть у нас в ADDR_ECC будет адрес на первый байт в двойном слове, а это как раз и есть 8 адресов, т.к. $16 \text{ адресов}\cdot 8 \text{ битов в байте} = 128 \text{ битов}$

> [!caution] 
>  Если ECCC или ECCD установлены, то ADDR_ECC и BK_ECC не будут обновляться даже если появится новая ошибка при чтении/записи. Регистр FLASH_ECCR обновляется только если ECCC и ECCD очищены

# Программирование и очистка Flash
После сброса запись в регистр управления Flash (FLASH_CR) запрещена, чтобы защитить Flash-память от возможных нежелательных операций, например, из-за электрических помех. Для разблокировки этого регистра используется следующая последовательность:

1) Записать KEY1 = 0x45670123 в регистр ключа Flash (FLASH_KEYR).
2) Затем записать KEY2 = 0xCDEF89AB в регистр FLASH_KEYR. То есть эти два ключа нужно записать последовательно

> [!important] 
> Для чтения эту последовательность проделывать не нужно. Чтение работает без использования FLASH_CR соответственно её можно делать в любой момент. 

Любая неправильная последовательность блокирует регистр FLASH_CR до следующего системного сброса. В случае неправильной последовательности ключей обнаруживается ошибка шины, и генерируется прерывание Hard Fault.

Регистр FLASH_CR может быть снова заблокирован программно путем установки бита LOCK в регистре FLASH_CR.

> [!caution] 
>  В регистр FLASH_CR нельзя записать, когда установлен бит BSY в регистре состояния Flash (FLASH_SR). Любая попытка записать в него при установленном бите BSY приводит к задержке шины AHB до тех пор, пока бит BSY не будет сброшен.

> [!attention] 
>  Операции записи в FLASH память и стирание страниц осуществляются с использованием внутреннего тактирования, то есть с помощью HSI, несмотря на то что как основной источник тактирования может быть HSE или PLL, для операций записи и стирания используется всегда HSI. 
>  Соответственно когда устанавливается бит START или происходит запись в адрес flash памяти, то внутренний осциллятор HSI включается автоматически. И автоматически выключается, когда запись заканчивается или сбрасывается START. Это отключение происходит за исключением случаев если HSI используется как основной генератор(т.е. включен HSION в регистре RCC_CR)
>  Операция чтения FLASH работает же по основному источнику тактирования микроконтроллера, не обязательно по HSI

## Очистка страницы
Чтобы стереть страницу, следуйте процедуре ниже:
1. Убедитесь, что не выполняется никакая операция с Flash-памятью, проверив бит BSY в регистре состояния Flash (FLASH_SR)
2. Проверьте и сбросьте все флаги ошибок программирования, возникшие из-за предыдущего программирования. В противном случае устанавливается бит PGSERR.
3. Установите бит PER и выберите страницу для стирания (PNB).
4. Установите бит START в регистре FLASH_CR.
5. Дождитесь, пока бит BSY не будет сброшен в регистре FLASH_SR.

## Запись данных
> [!caution] 
>  Запись в адрес который уже был запрограммирован невозможна аппаратно, тебе просто не даст сделать это микроконтроллер. Исключение если ты записываешь данные только из 0. Любая иная попытка записи в уже запрограммированный адрес флеш вызовет ошибку PROGERR в FLASH_SR.
>  Это сделано по той причине, что как мы знаем про [работу ПЗУ](Память#Постоянное%20запоминающее%20устройства) операция записи может лишь перевести биты из состояния 1 в состояние 0. Соответственно если уже записаны какие-то данные, то перезапись может дать некорректные данные, т.к. у нас останутся 0 от прошлой записи.

Последовательность программирования Flash-памяти:
1. Убедитесь, что не выполняется никакая операция с основной Flash-памятью, проверив бит **BSY** в регистре состояния Flash (**FLASH_SR**).
2. Проверьте и сбросьте все флаги ошибок программирования, возникшие из-за предыдущего программирования. В противном случае устанавливается бит **PGSERR**.
3. Установите бит **PG** в регистре управления Flash (**FLASH_CR**).
4. Выполните операцию записи данных по желаемому адресу памяти, внутри основного блока памяти или области OTP. Можно программировать только двойное слово.
	- Запишите первое слово по адресу, выровненному по двойному слову.
	- Запишите второе слово
> [!help] 
>  Но можно сразу записывать двойное слово через uint64_t тип данных в си, допустим
>  ```c
>  *(uint64_t *)(address+i) = data_to_write;
>  ```
>  Если data_to_write это 64 битный тип данных.

5. Дождитесь, пока бит **BSY** не будет сброшен в регистре **FLASH_SR**.
6. Убедитесь, что флаг **EOP** установлен в регистре **FLASH_SR** (что означает успешное завершение операции программирования), и сбросьте его программно.
7. Сбросьте бит **PG** в регистре **FLASH_CR**, если больше нет запросов на программирование.

> [!caution] 
> Если пользователю нужно запрограммировать только одно слово, двойное слово должно быть дополнено значением стирания **0xFFFF FFFF**, чтобы автоматически запустить программирование. 
> Контрольная сумма **ECC** вычисляется на основе запрограммированного двойного слова. 

# Изменение эффективной памяти в линкере
```ld
/* Memories definition */
MEMORY
{
  CCMSRAM    (xrw)    : ORIGIN = 0x10000000,   LENGTH = 32K
  RAM    (xrw)    : ORIGIN = 0x20000000,   LENGTH = 128K
  FLASH    (rx)    : ORIGIN = 0x8000000,   LENGTH = 512K
}
```
В значении LENGTH что выделен звёздочками нужно поменять значение LENGTH, в зависимости от того сколько вы заняли памяти
