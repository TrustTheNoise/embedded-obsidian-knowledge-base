# Объявление и определение

Объявление говорит компилятору Си, что где-то в программе, возможно, что и в другом файле, есть определение, связанное с этим именем. Определение может быть и в этом же файле, может быть и сразу на той же строке, как например:
```c
uint32_t some_variable = 21309821; // объявляем some_variable и определяем для него значение 21309821
```

Можно добавить при объявление слово extern это будет говорить компилятору, что определение для этого ключевого слова может находится в другом файле.
> [!important] 
> Без `extern` объявление переменной будет считаться **новым определением** 

> [!hint] 
>  Для объявлений функции можно не делать extern, компилятор всегда будет смотреть во всех файлах определение

Определение связывает имя которое было в объявлении с реализацией этого имени, которая может быть как данными(т.е. просто переменной), так и кодом(т.е. функцией).
- Определение переменной приводит к тому, что компилятор выделяет под неё память и возможно заполняет каким-то начальным значением
- Определение функции приводит к тому, что компилятор генерирует код для этой функции

Для переменных, определение бывает двух типов:
- Глобальные переменные, которые существуют всё время существования программы (статическое размещение) и к которым обычно обращаются из многих функций
- Локальные переменные, которые существуют только во время выполнения функции, в которой они объявлены (локальное размещение) и доступные только внутри неё

> [!help] 
> «доступный» значит, что на переменную можно ссылаться по имени, которое связано с её определением.

Однако не всё так просто с ключевым словом [[static]], можешь посмотреть эту заметку чтобы узнать подробности.

> [!summary] 
>  Объявление это обещание компилятору, что где-то в программе существует определение.

# Что делает компилятор?

Работа компилятора си в том, чтобы превратить файл с кодом из понимаемого (иногда) человеком в нечто, понимаемое компьютером, то бишь бинарная версия ассемблерного кода. 

На выходе компилятор даёт объектный файл, который на платформе UNIX имеет обычно расширение .o, а на windows .obj. Содержимое объектного файла это, по сути, два типа объектов:
- Код, соответствующий определениям функций
- Данные, соответствующие глобальным переменным, определённым в файле 

При этом для каждых из этих объектов будет создаваться ассемблерная метка(если переменные предварительно инициализированы, то там же хранится и их значение).

Конечно сами объектные файлы будут не в виде ассемблерного, а просто бинарниками, т.е. переведённым на понятный машине ассемблер, вспоминай что ассемблер это понятные человеку машинные операции, которые сами по себе это набор 1 и 0.

Но даже в объектном файле можно увидеть метки, т.к. они хранятся как текст. Чтобы в этом убедиться, можно воспользоваться утилитой nm, которая показывает метки из объектных файлов. 
Создадим такой си файл с названием first.c
```c title:first.c
/* Это определение неинициализированной глобальной переменной */
int x_global_uninit;

/* Это определение инициализированной глобальной переменной */ 
int x_global_init = 1;

/* Это определение неинициализированной глобальной переменной,  
но к ней можно обратиться по имени только из этого же си файла */ 
static int y_global_uninit;

/* Это определение инициализированной глобальной переменной,  
но к ней можно обратиться по имени только из этого же си файла */ static int y_global_init = 2;

/* Это объявление глобальной переменной, которая существует  
где-то в другом месте(не обязательно в этом же файле) */ 
extern int z_global;

/* Это объявление функции, которая определена где-то в другом  
месте программы. Можно добавть служебное слово extern. 
Но это не имеет значения */ 
int fn_a(int x, int y);

/* Это определение функции, но так как она определена со  
словом static, то доступна только в этом же си файле */
static int fn_b(int x) 
{   
	return x+1; 
}

/* Это определение функции. Её параметтры рассматриваются
как локальные переменные */
int fn_c(int x_local) 
{
	// Это определение неинициализированной локальной переменной
	int y_local_uninit;
	// Это определение инициализированной локальной переменной
	int y_local_init = 3;
	/* Этот код ссылает на локальные и глобальные переменные 
	и функции по имени */
	x_global_uninit = fn_a(x_local, x_global_init);
	y_local_uninit = fn_a(x_local, y_local_init);
	y_local_uninit += fn_b(z_global);
	return (y_global_uninit + y_local_uninit);
}
```

Скомпилируем в объектный файл, и рассмотрим через nm
```shell
gcc -c first.c -o first.obj

nm main.obj
```
И мы получим подобный вид
![[Pasted image 20250912155122.png|350]]

.xdata это универсальная секция программы. Обычно туда компилятор закидывает все метки, что объявленные что не объявленные, чтобы потом [Линкер](#Что%20делает%20Линкер?) расставил все эти метки в нужные места уже при создании исполняемого файла.

- Класс U означает неизвестный (unknown), или заглушку, как было сказано выше. Всего два таких объекта: fn_a и z_global
- Класс t или T обозначает, что код определён – t локально или T – это статическая функция.
- Класс d и D обозначают инициализированную глобальную переменную, d – локальную, D – не локальную. В исполняемом файле обычно находятся в .data
- Для неинициализированных глобальных переменных используется класс b, если статическая/локальная или B и C, если нет. В исполняемом файле обычно находятся в .bss

# Что делает Линковщик?

Итак как мы разобрались в [[#Объявление и определение]], объявление – это обещание компилятору, что где-то есть определение этой переменной или функции, и что работа линкера заключается в том, чтобы исполнять эти обещания.

Соответственно линкер смотрит все объектные файлы, находит для каждого объявления, определение, и собирает исполняемый файл, попутно закидывая все определения, все метки, из .xdata в соответствующие секции

| Секция     | Что кладёт компилятор/линкер                             | Когда заполняется                                 |
|-----------|---------------------------------------------------------|-----------------------------------------------------------|
| `.text`   | Машинный код функций, литералы только для чтения           | Всегда. Идёт во флэш/ROM.                                 |
| `.rodata` | Константы (`const`), строковые литералы                   | При компиляции. Тоже во флэш/ROM.                         |
| `.data`   | Глобальные/статические переменные **с начальным значением** | Образ значений хранится во флэше, копируется в RAM при старте. |
| `.bss`    | Глобальные/статические переменные **без инициализации**     | RAM очищается нулями во время старта.                     |

Для иллюстрации этого вот вам ещё один си файл в дополнение к первому c названием main.c

```c title:main.c
/* Инициализированная глобальная переменная  */ 
int z_global = 11; 
/* Вторая глобальная переменная с именем y_global_init,  
но обе они статические */ 
static int y_global_init = 2; 
/* Объявление ещё одной глобальной переменной */ 
extern int x_global_init;

int fn_a(int x, int y) 
{   
	return(x+y); 
}  

int main(int argc, char *argv[]) 
{
	const char *message = "Hello, world";
	return fn_a(11,12); 
}
```

Скомпилируем в объектный файл и этот файл, и соберём оба файла в один exe
```shell
gcc -c main.c -o main.obj

gcc -o out.exe .\main.obj .\first.obj
```

При просмотре через nm получится что-то такое(у меня не получилось но на сайте получилось)

Теперь как можно заметить те самые глобальные инициализированные данные были вложены в .data сегмент, неинициализированные глобальные переменные помещены в .bss а функции расположились в .text. 

Причём изначальные данные для .data загружаются из жёсткого диска(ну откуда мы и запускаем код). А .bss ядро ОС(или startup код в микроконтроллере) заполняет нулями, прям всю секцию .bss

## Роль ОС в процессе линковки

Так что же в этот момент происходит, как компилируемые файлы становятся исполняемыми? Мы написали сам код начинающийся с функции main. Однако стандарт си требует некоторую инициализацию, перед тем как начать выполнять код.

Так например данные о инициализированных глобальных переменных хранятся в ПЗУ, и соответственно их ОС должно загружать в сегмент `.data`. А неинициализированные данные заполняются нулями. Т.е. есть сегмент для неинициализированных данных `.bss` в exe расписаны метки для каждой из этих данных, соответственно ОС переносит эти метки в `.bss`, а сам сегмент `.bss` должен заполнять нулями.

Стек и кучу ОС тоже должна инициализировать перед выполнением программы. То есть она должна выделить место в своей RAM для стека+кучи и при этом выделить указатель для стека на самый верх этой выделенной памяти.

- Локальные переменные располагаются на участке памяти, известном как стек, который растёт и уменьшается, когда начинает исполняться или заканчивает работу функция
- Динамическая память выделяется на участке, известно как куча; выделением заведут функция malloc

Всё это должно происходить до начала нашей программы, которая начинается с main. Поэтому линковщик автоматически подтягивает стартовые файлы, которые всё это настраивают: (`crt1.o`), а также подтягивает стандартную библиотеку С.

В этих стартовых файлах есть реальная стартовая точка, которую уже ожидает ОС, это функция `_start` из которой вызывается код инициализации `__libc_start_main` который уже вызывается функция main, а в конце функции вызывает системный вызов exit, который принимает как раз таки возвратный код main.

Все системные функции (`sbrk`, `_write`, `_close`, `_read` и т. д.) будут требоваться линковщиком. Т.к. функция `_exit` вызывает функцию `fflush(stdout)`, или может выдать ошибку, а для этого нужен `error()` который вызывает `_write`. То есть кажется, что программа может быть очень простая, вплоть до
```c
int main(void)
{
    return 0;
}
```

То есть кажется что программка очень маленькая. И если рассматривать только её скомпилированный вариант объектного файла, то так оно и будет.
![[Pasted image 20250915205948.png|450]]
![[Pasted image 20250915211325.png|450]]

Однако во время линковки, когда мы будем создавать исполняемый файл нам уже потребуется правильно старта и конца программы, поэтому тут подключаются некоторые функции из стандартной библиотеки С, хотя вроде как мы их напрямую в нашем коде и не используем. А эти функции из стандартной библиотеки уже будут требовать системные вызовы, которые уже описаны самой ОС. Поэтому уже после линковки размер сильно вырастет
![[Pasted image 20250915211435.png]]

Все эти функции инициализации тогда же стека `_init` которая уже вызывает функции из стандартной библиотеки типа `__libc_csu_init` которая выполняет инициализацию всего, что должно быть готово до вызова, а также начальной точки `_start`, вместе с функцией из стандартной библиотеки которая вызывается после завершения программы `__libc_csu_fini` можно увидеть в том же самомо выводе символов из полученного исполняемого файла в [[#Что делает Линковщик?]]


![[Pasted image 20250912162127.png|500]]

И вот как раз `__libc_csu_init` и `__libc_csu_fini` будут вызывать другие системные вызовы, которые и будет требовать линковщик у ОС.

> [!summary] 
> Соответственно роль ОС это предоставление системных вызовов, которые позволят коду в Си делать нужную начальную инициализацию. 

Однако это ведь не всё, это только создание самого исполняемого файла. Теперь же нам нужно его запустить.

# Запуск исполняемого файл

Итак мы получили код, который естественно хранится в [ПЗУ](Память#Постоянное%20запоминающее%20устройства), а значит при запуске нам нужно эту программу перенести в RAM. Кусок памяти под программу называется сегментом кода или текстовым сегментом и обозначается как `.text`. Её нам и нужно перенести в RAM для работы с ней
> [!caution] 
> Современная ПЗУ стала довольно быстрой, поэтому сегмент `.text` часто не копируется заранее в RAM перед запуском.  
Вместо этого MMU (Memory Management Unit) создаёт отображение этого сегмента в виртуальное адресное пространство процесса.  
Это часть довольно сложного механизма [виртуальной памяти](Виртуальная%20память), которая обеспечивает изоляцию процессов, безопасность и эффективное использование памяти.

<span style="color:red">ОС сама выбирает место куда расположить все сегменты, в том числе .text!</span> Это важно осознать, т.к. сам линкер помещает начало программы в свой адрес, а уже ОС закидывает эти данные куда ему удобнее. Этот адрес куда линкер будет закидывать сегмент .text написан в стандартном линкер файле, который поставляется вместе с линкером gcc: `ld.exe`

Его даже можно просмотреть если при линковке выставить опцию 
```bash
ld --verbose
```

В Windows там можно найти такую строчку
```ld
  .text  __image_base__ + ( __section_alignment__ < 0x1000 ? . : __section_alignment__ ) :
```

Если честно хуй поймёшь какой адрес выбирается на Windows, потому что того же `__image_base__` не найдёшь в линкер скрипте...

А в Linux
```ld
 PROVIDE (__executable_start = SEGMENT_START("text-segment", 0x400000)); 
 . = SEGMENT_START("text-segment", 0x400000) + SIZEOF_HEADERS;
```

Тут куда более понятно, то есть начало для .text будет с 0x400000 + SIZEOF_HEADERS

И соответственно уже при запуске программы, ОС будет брать данные о коде из .text сегмента начиная с этих адресов.

Вся эта работа осложняется для микроконтроллеров, ведь у них нет никакой ОС, которая выдаст все эти системные вызовы, даст возможность правильно инициализировать все сегменты до выполнения программы и т.д. Про запуск кода на микроконтроллере можно прочитать тут [[Как микроконтроллер запускает код]]
