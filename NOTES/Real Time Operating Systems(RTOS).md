# Управление задачами

Операционная система обеспечивает абстракцию параллельно работающих процессов и потоков. Фактически в системах с одним процессором одновременно может быть запущен только один поток. 
Поэтому операционная система лишь чередует приложения для параллельного выполнения. Пока запущенный поток выполняется, все остальные ждут в очереди до следующего переключения задачи.

Существуют два основных метода переключения задачи:
- **Cooperative**(совместная) модель, где переключение задачи всегда является добровольным действием. Задачи добровольно отдают управление ядру, когда завершают свою работу или явно вызывают функцию переключения
- **Preemption**(вытясняющая) модель, которая требует, чтобы ядро периодически прерывало задачи в произвольный момент их выполнения, сохраняло состояние текущей задачи и возобновляло выполнение следующей задачи в очереди. Ядро RTOS должно прерывать выполнение задачи в любой момент, если: 
    - Истек квант времени
    - Появилась задача с более высоким приоритетом.

Переключение выполняемой задачи на низком уровне представляет собой сохранение значений регистров ЦП в ОЗУ и загрузку из памяти значений следующей задачи, выбранной для выполнения. Эта операция более известна как **context switching** и лежит в основе системы планирования.
На низком уровне, все потоки, или же **задачи** хранятся в **Task Control Block**(далее в тексте **TCB**) в котором как раз и хранится весь нужный контекст задачи:
- Указатель стека задачи(SP)
- Сохранённые значения регистров на момент context switching(`R0-R12`, `LR`, `PC`, `xPSR`)
- Дополнительные метаданные (приоритет, состояние, идентификатор).

## Task Control Block
Задачи представлены в системе в виде блочной структуры. Этот объект содержит всю информацию, необходимую планировщику для постоянного отслеживания состояния задачи. 

Каждый блок задачи может содержать указатель на функцию запуска, которая определяет начало кода, выполняемого при порождении задачи, и набор необязательных аргументов. Задачи могут быть определены на этапе компиляции и запущены после инициализации ядра, либо создаваться и завершаться динамически во время работы системы.

Каждая задача имеет указатель на функцию запуска, определяющую начало выполняемого кода, и может принимать набор аргументов. Для хранения локальных переменных и информации о вызовах функций каждой задаче необходим собственный стек выполнения. Ядро RTOS выделяет отдельную область RAM под стек каждой задачи, обеспечивая изоляцию контекста выполнения: при прерывании задачи регистры сохраняются в выделенной области памяти.
Как было сказано в [[#Управление задачами]], указатель стека для конкретной задачи хранится в структуре TCB при context switching.

Запуск с отдельными стеками требует, чтобы часть памяти была зарезервирована заранее и связана с каждой задачей. В простейшем случае все задачи, использующие стек одинакового размера, создаются до запуска планировщика. Область памяти, используемая для областей стека, может быть определена в [[Linker script]]. А нужные статически выделенные области могут выделиться до выполнения программы. 
Но более продвинутые планировщики поддерживают динамическое выделение памяти для новых задач во время работы системы, позволяя создавать процессы без остановки планировщика.

### Создание простого TCB

Итак в STM32G4 есть две основные области RAM: обычная [SRAM](Память#Статическое%20ОЗУ(SRAM)) и [CCM SRAM](Память#Core-Coupled%20Memory%20SRAM(CCM%20SRAM)). CCM SRAM напрямую связана с ядром процессора и обеспечивает быстрый доступ, поэтому она идеально подходит для размещения стека выполнения задач в RTOS. Часто всю область CCM SRAM резервируют исключительно под стеки задач, чтобы минимизировать задержки при переключении контекста.

Обычная SRAM используется для хранения секций программы, таких как `.data` и `.bss`, а также под динамическую память (кучу), необходимую приложению. Таким образом, разделение памяти позволяет эффективно управлять ресурсами и ускоряет работу ядра и планировщика.

Тогда конфигурация памяти, используемая для выделения задачам отдельных стеков выполнения будет выглядеть как-то так:
![[Pasted image 20251018212257.png|850]]

То есть мы расположим указатель задачи самого ядра RTOS в конце области CCM SRAM, а уже остальные задачи будем помещать относительно начала области CCM SRAM

Для того чтобы правильно разметить наши области памяти нужно изменить linker script подобным образом
```ld


MEMORY
{
  CCMSRAM    (xrw)    : ORIGIN = 0x10000000,   LENGTH = 32K
  SRAM    (xrw)    : ORIGIN = 0x20000000,   LENGTH = 128K
  FLASH    (rx)    : ORIGIN = 0x8000000,   LENGTH = 512K
}

_end_stack = ORIGIN(CCMSRAM) + LENGTH(CCMSRAM);
stack_space = ORIGIN(CCMSRAM);
_start_heap = _end /* _end defined in .bss section */


SECTIONS
{
    /* The program code and other data into "FLASH" Rom type memory */
    .text :
    {
    . = ALIGN(4);
    *(.text)           /* .text sections (code) */
    *(.text*)          /* .text* sections (code) */
    *(.glue_7)         /* glue arm to thumb code */
    *(.glue_7t)        /* glue thumb to arm code */
    *(.eh_frame)
    
    KEEP (*(.init))
    KEEP (*(.fini))
    
    . = ALIGN(4);
    _etext = .;        /* define a global symbols at end of code */
    } >FLASH
  
    /* Constant data into "FLASH" Rom type memory */
    .rodata :
    {
        . = ALIGN(4);
        *(.rodata)         /* .rodata sections (constants, strings, etc.) */
        *(.rodata*)        /* .rodata* sections (constants, strings, etc.) */
        . = ALIGN(4);
    } >FLASH
      
    /* Used by the startup to initialize data */
    _sidata = LOADADDR(.data);

    
    /* Initialized data sections into "RAM" Ram type memory */
    .data :
    {
    . = ALIGN(4);
    _sdata = .;        /* create a global symbol at data start */
    *(.data)           /* .data sections */
    *(.data*)          /* .data* sections */
    *(.RamFunc)        /* .RamFunc sections */
    *(.RamFunc*)       /* .RamFunc* sections */
    
    . = ALIGN(4);
    _edata = .;        /* define a global symbol at data end */
    
    } >SRAM AT> FLASH

    .bss :
    {
        _start_bss = .;
        *(.bss*)
        *(COMMON)
        . = ALIGN(4);
        _end_bss = .;
        _end = .;
    } > SRAM

}
```

Так что перейдём к коду ядра RTOS. 
В коде ядра RTOS стеки выполнения задач располагаются относительно начала CCM SRAM.
![[Pasted image 20251018212257.png|850]]
Чтобы корректно выделять области под каждый стек, ядру необходимо знать адрес начала CCM SRAM, который мы в линкер скрипте определили меткой `stack_space`, а также размер стека для каждой задачи.
```c
extern uint32_t stack_space;    // метка начала CCM SRAM из .ld скрипта
#define STACK_SIZE     (256U)   // размер стека в байтах для каждой задачи в TCB
```

Простая структура блока задач может быть объявлена следующим образом:
```c
typedef enum
{
    TASK_WAITING,
    TASK_READY,
    TASK_RUNNING,
}task_states;

#define TASK_NAME_MAXLEN 16

typedef struct {
    char name[TASK_NAME_MAXLEN]; // Имя задачи для удобства отладки и идентификации
    int id;                      // Уникальный идентификатор задачи
    task_states state;           // Текущее состояние задачи
    void (*start)(void *arg);    // Указатель на функцию, которая выполняет задачу
    void *arg;                   // Указатель на данные, передаваемые функции задачи
    uint32_t *sp;                // Указатель на текущий stack pointer задачи
}task_block;
```

Для хранения всех блоков задач используется глобальный массив. Глобальный индекс позволяет отслеживать количество созданных задач и вычислять позицию в памяти относительно идентификатора текущей задачи.
```c
#define MAX_TASKS 8
static task_block TASKS[MAX_TASKS];
static int n_tasks = 1;
static int running_task_id = 0;
#define kernel TASKS[0]
```
Первый элемент массива зарезервирован для блока задач ядра, которое является запущенным в данный момент процессом.
Чтобы создать задачу мы будем использовать функцию task_create где будем указывать имя, функцию которую и должна выполнять задача и аргументы для этой функции.
Для статической конфигурации с предопределенным количеством задач создание новой задачи делается при инициализации ядра, но более продвинутые планировщики могут позволить нам выделять новые блоки управления для порождения новых процессов во время выполнения, пока планировщик работает.
Итак функция task_create будет выглядеть так:

```c
task_block* task_create(char* name, void (*start)(void* arg), void* arg)
{
    struct task_block *t;
    int i;
    if (n_tasks >= MAX_TASKS)
    {
        return NULL;
    }
    t = &TASKS[n_tasks];
    t->id = n_tasks;
    n_tasks += 1;
    
    for (i = 0; i < TASK_NAME_MAXLEN; i++) {
        t->name[i] = name[i];
        if (name[i] == 0)
        break;
    }
    
    t->state = TASK_READY;
    t->start = start;
    t->arg = arg;
    // Вычисляем указатель стека для задачи:
    // 1. &stack_space — адрес начала области CCM SRAM, выделенной под стеки всех задач
    // 2. n_tasks * STACK_SIZE — смещение от начала области для текущей задачи
    //    каждая новая задача получает отдельный блок стека
    // 3. (uint8_t*) — приведение к типу указателя на байт для корректного обращения к памяти
    t->sp = (uint8_t*)((&stack_space) + n_tasks * STACK_SIZE);
    task_stack_init(t);
    return t;
}
```

В `task_stack_init` необходимо подготовить стек только что созданной задачи так, чтобы она могла участвовать в **context switching** так же, как уже выполняющиеся задачи. Как будет объяснено в главе [[#Context Switching]], при переключении задач сохраняются все регистры переключаемой задачи и загрузка из стека регистров следующей задачи. 

Регистры `r0–r3`, `r12`, `LR`, `PC` и `xPSR` автоматически сохраняются при входе в прерывание через NVIC, тогда как остальные регистры должны сохраняться вручную.

Нам же отсюда важно понять, что даже для новой задачи нужно правильно разместить все регистры в стеке, чтобы при выгрузке из стека она корректно восстанавливалась при context switching. Значения этих регистров могут быть произвольными, но их наличие в стеке обязательно для корректной работы RTOS. Таким образом, `task_stack_init()` формирует стек с «мусорными» значениями регистров, обеспечивая полноценное переключение контекста.
```c
typedef struct {
    uint32_t r0, r1, r2, r3, r12, lr, pc, xpsr;
} stack_frame;

typedef struct {
    uint32_t r4, r5, r6, r7, r8, r9, r10, r11;
} extra_frame;

static void task_stack_init(struct task_block *task)
{
    struct stack_frame *task_stack_frame;
    // Выделяем место под основной стековый кадр задачи
    task->sp -= sizeof(struct stack_frame);
    
    // Инициализация регистров, которые будут автоматически 
    // восстановлены при выходе из прерывания context switching
    task_stack_frame = (struct stack_frame *)(task->sp);
    task_stack_frame->r0 = (uint32_t) t->arg;
    task_stack_frame->pc = (uint32_t) t->start;
    task_stack_frame->lr = (uint32_t) task_terminated;
    task_stack_frame->xpsr = (1 << 24);
    
    // Выделяем место под дополнительные регистры, 
    // которые сохраняются вручную
    task->sp -= sizeof(struct extra_frame);
}
```

После восстановления контекста обработчик исключений автоматически восстанавливает регистры из стекового кадра, который для только что созданной задачи был заранее сформирован. Регистры задачи при создании инициализируются так, чтобы при первом переключении на задачу процессор корректно начал её выполнение.

- **Program counter** указывает на функцию задачи, куда система перейдёт при первом запуске.
- **R0-R3** могут содержать аргументы функции задачи. Но в нашем случае все задачи вида `void task(void* arg)` поэтому у нас только один аргумент, мы его и записываем в r0, остальные регистры остаются неинициализированы
- **Execution program status register xPSR** должен быть запрограммирован так, чтобы в бите 24 был установлен только обязательный флаг набора инструкций Thumb;
- **Link register LR** содержит указатель на процедуру, которая вызывается при возврате из функции задачи. В нашем случае задачам не разрешается возвращаться из функции запуска, поэтому функция `task_terminated` – это просто бесконечный цикл, и это считается системной ошибкой. В других случаях, если задачам разрешено завершаться, функция может играть роль общей точки выхода для задач, чтобы выполнять операции очистки, необходимые при возврате из функции запуска.
- **R4-R11** регистры не сохраняются автоматически при прерываниях, так как используются для локальных вычислений в функциях. Поскольку задача только создаётся и ещё не выполнялась, их значения в стеке можно оставить произвольными, а не инициализировать вручную 0.

После создания начального кадра стека задача начинает участвовать
в многозадачности и может быть выбрана планировщиком в любое время для возобновления выполнения из того же состояния, как и остальные неработающие задачи

## Context Switching

В RTOS процедура **context switching** заключается в сохранении состояния текущей задачи и восстановлении состояния следующей. Состояние включает регистры процессора, которые сохраняются в стеке выполняемой задачи. В современных микроконтроллерах операция context switching выполняется обычно в обработчике прерываний, поскольку только прерывание может безопасно прервать выполнение текущей задачи и передать управление ядру.

О том, какие прерывания используются в STM32 для переключения задач, будет рассказано в [[#Планировщик]], но важно понимать: в RTOS **context switching всегда происходит через прерывание**.

И что интересно в Cortex-M4 при входе в любой обработчик прерывание контроллер  NVIC автоматически сохраняет часть регистров, формируя stack frame, чтобы после завершения обработчика можно было восстановить контекст прерванного кода. Этот кадр стека содержит регистры **R0–R3, R12, LR, PC и xPSR** в определённом порядке, показанном на рисунке ниже:
![[Pasted image 20251019133234.png]]

И это как раз удобно для RTOS: при переключении контекста через прерывание часть регистров уже сохраняется автоматически. Но NVIC не захватывает другую половину регистров ЦП – с R4 по R11. По этой причине для успешного завершения переключения контекста ядро, намеревающееся заменить работающий процесс RTOS в обработчике PendSV или SVCall, должно сохранить дополнительный кадр стека, содержащий значение для этих регистров, и восстановить для следующей задачи непосредственно перед возвратом из обработчика. 

У Thumb-2 есть все необходимые ассемблерные инструкции для размещения значений смежных регистров ЦП в стеке и возврата их на место.
Поэтому функции сохранения и восстановление дополнительных регистров будет выглядеть так:
```c
static void __attribute__((naked)) store_context(void)
{
    asm volatile("mrs r0, msp");
    asm volatile("stmdb r0!, {r4-r11}");
    asm volatile("msr msp, r0");
    asm volatile("bx lr");
}

static void __attribute__((naked)) restore_context(void)
{
    asm volatile("mrs r0, msp");
    asm volatile("ldmfd r0!, {r4-r11}");
    asm volatile("msr msp, r0");
    asm volatile("bx lr");
}
```
> [!help] 
> В Cortex-M4 существует два указателя стека: **MSP** (Main Stack Pointer) и **PSP** (Process Stack Pointer). MSP используется ядром и обработчиками прерываний, тогда как PSP — для выполнения задач пользовательского уровня, например, потоков RTOS.
> 
> Регистр **SP** является псевдонимом текущего активного указателя стека: все инструкции работы со стеком (PUSH, POP, CALL, RET и т. д.) используют именно SP, который в зависимости от режима выполнения указывает либо на MSP, либо на PSP. Переключение между ними управляется самим ядром, когда оно переходит из режима задачи в режим обработчика и обратно.

> По этой причине компилятор не должен вмешиваться в работу функции, сохраняющей контекст регистров. В ARM-функциях регистры **R4–R11** используются для локальных вычислений в функции, и компилятор автоматически сохраняет **только используемые** регистры в функции в прологе функции и восстанавливает их в эпилоге.
> Однако мы не можем предсказать, какие именно регистры будут использованы — это может быть весь диапазон **R4–R11** или только часть. Поэтому пролог и эпилог нужно отключить, чтобы вручную точно сохранить все регистры в кадр стека задачи.
> Для того чтобы сказать компилятору GCC не генерировать пролог и эпилог используется `__attribute__((naked))`

## Планировщик

В современных микроконтроллерах операция [[#Context Switching]] выполняется обычно в обработчике прерываний, поскольку только прерывание может безопасно прервать выполнение текущей задачи и передать управление ядру.

В stm32g4 с ядром Cortex-M4 существуют два ключевых системных исключения, специально предназначенные для поддержки context switching:

- **PendSV(Pending Supervisor Call)**

Это низкоприоритетное исключение, которое вызывается аппаратно установкой специального бита PENDSVSET в регистре **SCB_ICSR**(stm32 programming manual). Оно не запускает прерывание PENDSVSET немедленно, а ожидает окончания всех прерываний с более высоким приоритетом, что делает его идеальным для выполнения операций переключения контекста, не мешающих обработке более критичных прерываний. В RTOS PendSV обычно используется ядром для принудительного вытеснения(preemption метод переключения) текущей задачи и переключения на следующую готовую к выполнению.

Пример: если задача с низким приоритетом выполняется, а в систему поступила задача с более высоким приоритетом, ядро устанавливает бит PendSV. После завершения текущего активного прерывания с более высоким приоритетом Cortex-M вызывает PendSV, который сохраняет контекст текущей задачи и восстанавливает контекст новой задачи.

- **SVCall (Supervisor Call)** 

Это исключение, которое вызывается программно ассемблерной инструкцией SVC. 
 - [ ] #todo Найти есть ли в CMSIS функция для инструкции SVC или нужно вызывать inline ассемблером

Он используется для безопасного выполнения системных операций, которые не могут выполняться напрямую из пользовательской задачи, например: создание задач, работа с мьютексами, очередями сообщений или приостановка задачи.

Например если операция требует ожидания (например, задача sleep на 100 мс), то она вызывает инстуркцию SVC и запрашивает у ядра задачу sleep на 100 мс. Вызывается SVC_Handler, который принадлежит ядру, где ядро помещает задачу в список ожидания и **вытесняет** её: сохраняет её контекст (регистры, стек) и переключает процессор на другую задачу. Когда условие ожидания выполнится (таймер истечёт, мьютекс освободится и т.д.), ядро снова пометит задачу как `READY` и вернёт ей управление.


### Реализация планировщика

Итак Основной компонент планировщика представляет собой обработчик исключений, связанных с событиями прерывания системы, таких как PendSV и SVCall. Как уже было сказано PendSV исключение вызывается аппаратно установкой специального бита PENDSVSET в регистре **SCB_ICSR**. Воспользуемся этим исключением для вызова планировщика. Создадим простой макрос для переключения контекста
```c
#define schedule() SCB_ICSR |= SCB_ICSR_PENDSVSET_Msk
```

Вызов schedule приведут к исключению PendSV и соответственно вызову обработчика PendSV, где мы можем создать context switching

```c
void __attribute__((naked))PendSV_Handler(void)
{
    // Сохраняем контекст текущей задачи
    store_context();
    
    // Сохраняем указатель стека текущей задачи в её TCB
    asm volatile("mrs %0, msp" 
        : "=r"(TASKS[running_task_id].sp)
    );
    
    // Меняем состояние текущей задачи на READY 
    // (готова к выполнению, но не активна)
    TASKS[running_task_id].state = TASK_READY;
    // Выбираем следующую задачу (циклически)
    running_task_id = (running_task_id + 1) % n_tasks;
    
    // Отмечаем следующую задачу как RUNNING
    TASKS[running_task_id].state = TASK_RUNNING;
    
    // Загружаем стековый указатель следующей задачи
    asm volatile("msr msp, %0"
        :
        :"r"(TASKS[running_task_id].sp)
    );
    
    // Восстанавливаем контекст следующей задачи
    restore_context();
    
    // Подготавливаем возврат из прерывания
    // LR = 0xFFFFFFF9 — специальное значение EXC_RETURN для 
    // возврата в Thread mode с использованием MSP
    asm volatile("mov lr, %0" ::"r"(0xFFFFFFF9));
    // Возврат в следующую задачу
    asm volatile("bx lr");
}
```
> [!help] 
> Я думаю не понятно что за значение что мы выставляем в LR. 
> При входе в обработчик прерываний в регистр LR записывается не какой-то адрес, а специальный код EXC_RETURN, старшие 28 бит которого установлены в 1, а биты [3:0] содержат информацию, необходимую для выполнения операции возврата.
> 
> Однако в обработчике с атрибутом `__attribute__((naked))` компилятор не генерирует стандартный prologue/epilogue и не управляет LR автоматически. Поэтому мы сами записываем в LR значение `0xFFFFFFF9`. Что значит выйти с режимом Thumb, режимом потока и с использованием стека MSP
> Более подробно про EXC_RETURN можно прочитать в [[Прерывания#EXC_Return]]
