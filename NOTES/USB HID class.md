# Структура дескрипторов

Первым важным моментом для USB HID класса устройств является то, что HID класс не может быть определён at device descriptor level. То есть HID устройства определяются на уровне интерфейсов, поэтому для определения HID устройства используйте `bInterfaceClass` и `bInterfaceSubClass` поля в **Interface Descriptor**.

Для HID class вместе с interface descriptor USB устройство вместе с дескрипторами для endpoint также должен отправить HID descriptor, который состоит из:
- Report descriptor - описывает каждый фрагмент данных, который генерирует устройство, и то, что именно эти данные измеряют. Например, дескриптор отчётов определяет элементы, описывающие положение или состояние кнопок. 
    Report descriptor помогает драйверу понять как парсить входной report о состоянии USB HID устройства и куда направлять эти входные данные.
- Physical descriptor - являются необязательными дескурипторами которые предоставляют информацию о той части или частях человеческого тела, которые используются для управления элементами устройства

# Report descriptor

HID-стандартизаторам быстро стало ясно, что поле `bInterfaceSubclass` нельзя использовать для определения подкласса HID-устройства из-за большого и постоянно растущего числа устройств. Поэтому было решено, что каждое HID-устройство будет описываться через **Report descriptor**, который указывает тип устройства и определяет, какие данные оно передаёт.

То есть report descriptor определяет в каком формате данные из HID устройства будут отправляться по Interrupt IN endpoint. Так например если устройство - мышка, то данные будут отправляться в report'ах с данными о перемещении мыши и нажатиям кнопок. И то в каком формате будут передаваться эти данные, как обрабатывать эти данные и т.д. и определяет report descriptor/

## Report items
Report descriptor передаётся не как целая таблица, как другие USB-дескрипторы, а поэлементно, поскольку его размер может быть очень большим.

Каждый элемент, называемый **item**, представляет собой часть информации из report descriptor. Каждый item начинается с однобайтового префикса, который содержит **item tag**, **item type** и **item size**.
![[Pasted image 20251227212201.png|700]]

Существуют два типа items: 
- **short item** у которого поле `bSize` может быть от 0 до 4 байт.
- **long item** у которого `bSize` всегда равен 2, `bTag = 0b1111`, `bType = 0b11`; размер и тег item задаются в двух следующих байтах после префикса.
![[Pasted image 20251227212442.png|700]]

На стороне хоста работает **item parser**, который последовательно анализирует report descriptor, определяет тип каждого элемента и сохраняет их в **item state table**. Эта таблица — область памяти на компьютере, где накапливаются данные элементов report descriptor, после чего каждый элемент привязывается к соответствующей переменной (например, координатам осей или состоянию клавиш).

Элементы разделяются на несколько типов:
1. Main `bType=0` 

Создают **поле отчёта** — то, что устройство будет **отправлять хосту или получать от него**. Main item всегда применяется после того, как были заданы все необходимые параметры для этого поля (через Global и Local items). То есть только когда parser прошёл через все Global/Local элементы и собрал item state table, Main item фактически «создаёт» поле отчёта.

Примеры Main items:
- `Input` - данные, которые устройство отправляет на хост (например, положение осей, нажатие кнопок).
- `Output` - данные, которые хост отправляет на устройство (например, включение светодиода).
- `Feature` — данные, которые устройство может хранить, но пользователь напрямую не видит (например, настройка профиля).
- `Collection` - объединяет несколько Main items в логическую группу (например, джойстик, клавиатура).
- `End Collection` — закрывает коллекцию.

*Для понимания структуры каждого из этих main item обратитесь к главе 6.2.2.4 USB HID specification 1.11*

2. Global `bType=1`

Тип для параметров main item'ов которые будут сохраняться глобально в item state table. То есть когда мы уже закончим main item отправив  Input/Output/Feature item то параметры с типом global останутся в item state table и могут использоваться в следующих main item, либо могут быть перезаписаны.

Обязательные global item для каждого main item:
- `Usage Page` — тип устройства (например, Generic Desktop, Button)
- `Logical Min/Max` — минимальное и максимальное значения данных
- `Report Size` — размер одного поля в битах
- `Report Count` — количество одинаковых полей

> [!Опционально] 
>  - `Physical Min/Max` — реальные физические значения (например, мм, градусы)
>  - `Unit` / `Unit Exponent` — единицы измерения (мм, м, градусы)


 3. Local `bType=2`


Тип для параметров main item'а которые не сохраняются глобально в item state table. То есть когда мы уже закончим main item отправив  Input/Output/Feature item то параметры с типом local будут удалены из item state table и не могут быть использованы в следующих main item.

Обязательный local item:

- `Usage` — конкретная ось или кнопка (например, X, Y, Hat Switch, Button 1)

>[!Опционально]
>- `Usage Min/Max` — диапазон для массивов (например, кнопки 1…32)
>- `Designator Index/Min/Max` — идентификаторы аппаратных элементов
>- `String Index/Min/Max` — индексы строк для отображения имени элемента

Item state table мы определяем элементы report которые уже в runtime будет отправлять HID устройство. То в каком порядке мы определили элементы в report descriptor в таком порядке HID устройство и будет отправлять данные в runtime. Определение каждого элемента report происходит следующим образом:
1. Определяется назначение элемента в report'e. Назначение определяется через 
   - Usage Page - категорию устройства к которой будут относится элементы report'а. Например, категория **Generic Desktop Controls** включает устройства управления ПК: мыши, клавиатуры, джойстики, геймпады и т.д.
   - Usage — это конкретное назначение элемента внутри выбранной категории. Он определяет, что именно представляет этот элемент в репорте. Например если мы выставим Usage Page как Generic Desktop Controls то Usage может обозначать координаты X, Y, Z или клавиши на клавиатуре.
   
   Usage Page говорит «в какой области устройства мы работаем», а Usage говорит «какой конкретный элемент мы описываем». Это позволяет хосту точно понять, что означают данные, приходящие от HID-устройства.
2. Определяем параметры элемента. Это включает в себя такие параметры как:
    -  Диапазон значений, которые может принимать элемент (Logical Minimum / Maximum).
    - Размер элемента в репорте (Report Size), количество элементов (Report Count).
    - Тип данных и способ их интерпретации (Input, Output, Feature).
3. Созданием самого элемента через Main item

> [!example] 
>  Давайте рассмотрим как item state table будет работать на практике.
>  Например, item parser последовательно встречает несколько элементов из report descriptor.
>```
>    HID_USAGE_PAGE     ( HID_USAGE_PAGE_DESKTOP )
>    HID_USAGE          ( HID_USAGE_DESKTOP_X)
>    HID_LOGICAL_MIN    ( -127 )
>    HID_LOGICAL_MAX    ( 127 )
>    HID_REPORT_COUNT   ( 1 )
>    HID_REPORT_SIZE    ( 8 )
>    HID_INPUT    (HID_DATA | HID_VARIABLE | HID_ABSOLUTE)
>```
>  1. Global Item: `Usage page = Generic Desktop Controls` - парсер узнаёт, что следующие main item'ы будут принадлежать к категории desktop controls.
>  2. Local item: `Usage = HID_USAGE_DESKTOP_X` — parser узнаёт, что следующий Main item относится к оси X.
>  3. Global Items:
>      `Logical Min = -127`
>      `Logical Max = 127` 
>      `Report Count = 1`
>      `Report Size = 8`
>   Парсер сохраняет эти глобальные параметры в **item state table**, пока не встретит Main item. То есть USB HID устройство будет отправлять значение оси X одним сообщением(Report Count=1) на 8 бит(Report Size = 8) со значениями от -127 до 127
>   4. Main item: `Input` только после этого item создаётся поле отсчёта для X:
>  ```
>  X_axis:
>      logical_min = -127
>      logical_max = 127
>      size = 8
>  ```
>  Затем приходит следующие элементы:
>  ```
>    HID_USAGE          ( HID_USAGE_DESKTOP_Y)
>    HID_REPORT_COUNT   ( 1 )
>    HID_REPORT_SIZE    ( 8 )
>    HID_INPUT   (HID_DATA | HID_VARIABLE | HID_ABSOLUTE)
>  ```
> — создаётся поле для оси Y. Global items остаются прежними, мы же заново создаём только local items
>```
>    HID_USAGE          ( HID_USAGE_DESKTOP_Z)
>    HID_LOGICAL_MIN    ( 0 )
>    HID_LOGICAL_MAX    ( 255 )
>    HID_REPORT_COUNT   ( 1 )
>    HID_REPORT_SIZE    ( 8 )
>    HID_INPUT   ( HID_DATA | HID_VARIABLE | HID_ABSOLUTE )
>```
> — для оси Z обновляются и Global items (`Logical Min/Max`)
> 
> Таким образом теперь хост знает обо всех элементах которые будет отправлять USB HID устройство, таким образом во время runtime USB HID будет передавать примерно такие report:
> ```
> [ X_axis | Y_axis | Z_axis ]
> [  12    |  -45   |  200   ] 
> ```

> [!important] 
> Большинство устройств формируют отчёты (reports) как структуру, где все поля данных идут последовательно. 
> Однако некоторые устройства могут иметь несколько независимых структур отчётов на одном endpoint, каждая содержащая только часть данных. Например, клавиатура с тачпадом может отдельно отправлять данные о нажатиях клавиш и данные указателя через один endpoint.
> Для различения структур используется **Report ID** — однобайтовый идентификатор, который ставится в начале каждого отчёта. Если в Report Descriptor нет Report ID, считается, что есть только одна структура report.
> Если устройство имеет несколько отчётов, каждый report начинается с **одного байта идентификатора**, позволяя драйверу различать, например, данные клавиатуры и данные указателя.
